# EvoClaw — Unified System Design

**以组织进化为核心的 AI 管家系统**
Unified Design Document · 综合 v1–v4 + 参考文档 · 保密

---

## 0. 设计哲学与核心定义

### 0.1 三个核心词的精确定义

**进化 (Evolution)**
某个可测量指标在时间轴上显著改善，且这种改善来自系统内部的自动调整，而不是人工手动修改配置。用户手动添加插件、开发者更新代码、系统换底层模型——这些是升级，不是进化。

**组织 (Organization)**
多个 agent 按照明确的角色分工和通信协议协作，使整体能力**系统性地**超过任意单个 agent 的能力上限。关键词是"系统性"——不是偶尔，不是靠运气，而是在结构层面保证的能力放大。

**能力模块 / 插件 (Capability Module / Plugin)**
一个完整的合约包，包含：系统提示词（定义角色和行为边界）、工具集（被允许调用的外部工具）、性能基线（量化判断标准）、版本记录（每次进化的 diff 和原因）。插件不是一段代码，不是一个 API，而是这四个要素的组合体。

### 0.2 设计原则

| # | 原则 | 含义 |
|---|------|------|
| 1 | 可测量才叫进化 | "感觉更好"不是进化，是漂移。进化必须体现为预定义指标的统计显著改善。 |
| 2 | 进化有边界 | 什么可以自动修改、什么必须人工确认、什么永远不变——三类划分必须显式定义。 |
| 3 | 评估系统独立 | 评估进化结果的 Judge 必须与执行业务的 Critic 物理隔离。不能用被进化影响的指标来评估进化本身。 |
| 4 | 失败是一等公民 | 挑战集（Challenge Set）与历史成功日志同等重要。只用成功案例做回放是在优化幸存者。 |
| 5 | 资源约束驱动结构 | 没有资源压力，组织不会形成真实的结构选择。Token 预算、调用上限必须写进组织层。 |
| 6 | 进化可以休眠 | 一个健康的系统知道什么时候"足够好了"。进化引擎有明确的休眠条件。 |
| 7 | 失败驱动而非成功驱动 | 每次任务完成后运行反事实推理："如果失败了，最可能的原因是什么？" |
| 8 | 每个设计决策必须能回答"如果这里失败了，会发生什么？" | 如果回答是"系统崩溃"或"不知道"，这个设计就需要重做。 |

### 0.3 哲学演进

| 阶段 | 设计哲学 | 核心比喻 |
|------|----------|----------|
| v1/v2 | 用组织制造质量 | 手工作坊：每个工人理解全部流程，靠经验打磨产出质量 |
| v3 | 用标准制造规模 | 工厂流水线：标准化零件、标准化接口、标准化验收 |
| v4 | 用有机体制造未来 | 生命体：稳定的细胞结构 + 探索性的免疫系统 + 主动代谢 |
| 统一版 | 进化被制度化，组织被约束化，复杂度被预算化 | 自稳定操作系统：会探索、会压缩、会发布、会瘦身、会自我约束 |

## 1. 系统架构总览

EvoClaw 采用多层架构 + 横切工具化层的设计。各层之间严格单向依赖：上层读取下层输出，下层不感知上层存在。

```
┌─────────────────────────────────────────────────────────────────┐
│                  Facade（用户感知层）                              │
│              统一人格 · 统一语气 · 统一确认流程                      │
└──────────────────────────┬──────────────────────────────────────┘
                           │
┌──────────────────────────┼──────────────────────────────────────┐
│           Governance Kernel（治理内核 · 不可自动替换）               │
│       发布/灰度/回滚 · 权限裁决 · 冻结进化 · 宪法升级门禁           │
└──────────────────────────┬──────────────────────────────────────┘
                           │
  ┌────────────────────────┼────────────────────────────────┐
  │                        │                                │
  │  ┌─────────────────────┴──────────────────────────┐     │
  │  │        进化健康层 (Evolution Health)             │     │
  │  │  Meta Judge · Challenge Curator                 │     │
  │  │  Dormancy Controller · Entropy Monitor          │     │
  │  └────────────────────┬───────────────────────────┘     │
  │                       │                                 │
  │  ┌────────────────────┴───────────────────────────┐     │  ┌──────────────────────┐
  │  │          进化层 (Evolution)                      │     │  │ 工具化层 (横切)       │
  │  │  Observer · Evolver · Constitution              │     │  │ UMI · SLP            │
  │  │  Compiler (跨区独立)                             │     │  │ Safety Airbag        │
  │  └────────────────────┬───────────────────────────┘     │  │ Capability Matrix    │
  │                       │                                 │  │ Contract Validator   │
  │  ┌────────────────────┴───────────────────────────┐     │  └──────────────────────┘
  │  │          记忆层 (Memory)                         │     │
  │  │  Canonical Event Log (基础设施)                   │     │
  │  │  User Model · Module Memory                     │     │
  │  │  Organization Log · Challenge Set               │     │
  │  └────────────────────┬───────────────────────────┘     │
  │                       │                                 │
  │  ┌────────────────────┴───────────────────────────┐     │
  │  │        执行层 (Execution · 双区)                  │     │
  │  │  ┌─────────────┐    ┌─────────────┐            │     │
  │  │  │  稳定区      │    │  探索区      │            │     │
  │  │  │  Stable Zone │    │  Exploration │            │     │
  │  │  └─────────────┘    └─────────────┘            │     │
  │  │  Router(ε-greedy) · UMI Modules                 │     │
  │  │  Operational Critic · Synthesizer               │     │
  │  └────────────────────┬───────────────────────────┘     │
  └────────────────────────┼────────────────────────────────┘
                           │
              用户 (Telegram · 飞书 · 微信 · Discord)
```

### 各层职责一句话

| 层 | 一句话职责 |
|----|-----------|
| **Facade** | 用户永远只看到一个统一人格的管家，内部组织对用户完全透明 |
| **Governance Kernel** | 系统是否允许某个变更存在——发布/回滚/冻结/权限裁决 |
| **进化健康层** | 用独立标准评估进化结果是否真的变好了 |
| **进化层** | 读取健康数据，设计进化方案，运行 A/B 测试，编译新能力 |
| **记忆层** | 多粒度记忆存储，Canonical Event Log 作为不可变的事实基础设施 |
| **执行层** | 直接面向用户任务，双区架构（稳定区高效利用 + 探索区实验未知） |
| **工具化层** | 横切所有层，提供 UMI 接口标准、SLP 语义原语、Safety Airbag 安全工程 |

### 关键架构决策：Event Log 的定位

Canonical Event Log 作为记忆层的**核心基础设施组件**存在，而非独立的 Layer 0。所有行为记录为不可变事件，所有决策必须可追溯，所有插件调用必须有证据链。记忆层的其他组件（User Model、Module Memory、Organization Log、Challenge Set）是 Event Log 之上的**派生视图**。

这个混合方案的优势：
- 保持了 v1–v4 记忆层的概念完整性
- 获得了 gptv5 提出的不可变事件账本的可审计性
- 用户模型、模块记忆等仍然是可查询的结构化数据，不需要每次从原始事件重建

## 2. 角色定义与职责边界

每个角色只做一件事，并且明确定义它**不做什么**。这是防止角色漂移的唯一方式。

### 2.1 执行层角色

| 角色 | 唯一职责 | 不做什么 |
|------|----------|----------|
| **Router** (ε-greedy) | 解析用户意图，查询 Capability Matrix 做路由决策。以概率 (1-ε) 选择评分最高的模块，以概率 ε 随机探索 | 不执行任务，不修改输出，不理解模块内部实现 |
| **Capability Module** (UMI) | 在自己的能力域内执行任务并返回标准化结果 | 不路由其他模块，不评判自身输出质量 |
| **Operational Critic** | 在输出交付用户前评估质量，标记问题 | 不修改输出内容本身，不执行任务 |
| **Synthesizer** | 把多个模块的输出整合成统一回复 | 不重新执行任务，不做质量评判 |

**关键设计**：Critic 与 Capability Module 的激励对立——Module 优化任务完成率，Critic 优化错误发现率。这种内建张力保证系统不退化为回音壁。

### 2.2 记忆层（数据结构，非角色）

记忆层不是 agent，而是一套分层数据结构：

| 组件 | 说明 |
|------|------|
| **Canonical Event Log** | 不可变的 append-only 事件账本。所有行为、决策、状态变更都记录为事件。是一切派生数据的唯一真实来源。 |
| **Working Memory** | 当前会话的完整上下文。会话结束后压缩提炼，丢弃原始记录。 |
| **User Model** | 从交互中结构化提炼的用户偏好。永不存储原始对话。 |
| **Module Memory** | 每个能力模块的 RAG 知识库。包含领域知识和历史执行模式。 |
| **Organization Log** | 每次任务的执行摘要：模块版本、耗时、Critic 评分、用户反馈信号。 |
| **Challenge Set** | 失败/边界案例集合。进化层的核心评估数据源。 |

### 2.3 进化层角色

| 角色 | 唯一职责 | 不做什么 |
|------|----------|----------|
| **Observer** (含 Entropy Monitor) | 定期读取日志，生成结构化健康报告（含熵度量），标记偏离基线的维度 | 不提解决方案，不修改任何东西 |
| **Evolver** | 读取健康报告，设计进化方案，运行 A/B 测试，推送更新 | 不接触用户交互，不直接读日志（通过 Observer 间接感知） |
| **Constitution** | 定义所有角色都不能跨越的行为边界，以及什么级别的进化需要用户确认 | 宪法本身不可自动修改，只能用户授权 |
| **Compiler** | 监听探索区事件日志，聚类重复协作模式，生成 Pattern，编译为 UMI 模块或轻量工具，提交晋升申请 | 不执行业务任务，不直接修改稳定区，不触发进化 |

**Maintainer 替换规则**：Observer + Evolver 组合（Maintainer）是进化机制核心，**不可自动替换**。若 Maintainer 可以被自己提案替换，就产生自指循环——当其质量下降时，它提出的替换方案也是低质量的。Maintainer 的替换只能由用户通过显式操作触发，不走进化流程。

### 2.4 进化健康层角色

| 角色 | 唯一职责 | 不做什么 |
|------|----------|----------|
| **Meta Judge** | 独立评估进化结果。仅在 A/B 裁决阶段出现，不知道哪个是候选版本（盲测） | 不参与日常执行，不与 Operational Critic 共享任何上下文 |
| **Challenge Curator** | 主动构建和维护 Challenge Set（失败/边界案例），按 SLP 原语分类 | 不参与日常任务执行 |
| **Dormancy Controller** | 监控进化引擎是否应进入休眠或唤醒 | 不主动发起进化，只做状态控制 |

### 2.5 工具化层角色

| 角色 | 唯一职责 | 不做什么 |
|------|----------|----------|
| **Contract Validator** | 校验新模块的 UMI 契约完整性和 airbag_spec 合规性。纯机械校验，无 LLM 参与 | 不做语义理解，不做业务逻辑 |
| **Capability Matrix Manager** | 维护 Capability Matrix，记录所有模块在所有 SLP 原语上的历史质量评分 | 不路由任务，不评判质量 |
| **Pattern Registry** | 存储探索区提炼出的 Pattern 草稿，管理 Pattern 生命周期 | 不路由任务，不评判质量 |

### 2.6 用户感知层

| 角色 | 唯一职责 |
|------|----------|
| **Facade** | 统一人格输出、统一解释风险、统一请求确认、统一展示系统成长。内部组织对用户完全透明，用户永远只看到一个管家。 |

### 2.7 治理内核

| 角色 | 唯一职责 |
|------|----------|
| **Governance Kernel** | 发布/灰度/回滚决策、权限裁决、冻结进化、宪法升级门禁、风险分级。不做业务，只管"系统是否允许某个变更存在"。任何自动生成的插件必须经过它。**不可自动替换**。 |

## 3. 核心数据结构

### 3.1 Task

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | UUID | 不可变 | 全局唯一标识，用于追溯和回放 |
| `goal` | string | 可验收 | 目标描述，必须能被 `acceptance` 字段验证 |
| `acceptance` | AcceptanceCriteria | 必填 | 验收标准，自动化或人工，必须可执行 |
| `budget.tokens` | integer | 硬上限 | 本次任务所有 agent 可消耗的总 token 上限 |
| `budget.tool_calls` | integer | 硬上限 | 工具调用次数上限，超出 Governor 强制收敛 |
| `budget.rounds` | integer | 硬上限 | 内部轮次上限，防止无限循环 |
| `budget.time_sec` | integer | 软上限 | 超时触发降级，不强制终止 |
| `risk_level` | enum L0–L4 | 必填 | 决定权限门控级别和是否需要多签 |
| `permissions` | PermissionSet | 白名单制 | 允许访问的资源范围，默认全部拒绝 |
| `state` | TaskState | 状态机控制 | 见 3.3 状态机 |
| `artifacts` | Artifact[] | 任务产出 | 代码 diff、测试报告、摘要等可验证产物 |
| `provenance` | string[] | 可选 | 本任务依赖的历史任务 id 链 |
| `challenge_flag` | boolean | 默认 false | 为 true 时进入 Challenge Set |
| `capability_requirements` | SLPrimitive[] | 可选 | 完成本任务需要的原语列表 |
| `preferred_modules` | string[] | 可选 | 用户或进化历史指定的首选模块 |
| `airbag_required` | AirbagLevel | 可选 | 本任务要求的最低安全工程级别 |

**资源预算的结构性意义**：没有资源约束，组织不会形成结构选择压力。Token 预算有限 → Router 倾向精准路由；Tool Call 预算有限 → Planner 生成更精准的子任务；进化预算有限 → Evolver 优先处理影响最大的问题。

### 3.2 Message

所有 agent 间通信遵循统一结构化格式，保证零信任可验证。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `msg_id` | UUID | 不可变 | 全局唯一，用于追溯消息来源链 |
| `source` / `destination` | RoleID | 必填 | 发送方和接收方角色标识 |
| `task_id` | UUID | 必填 | 所属任务，关联 Task 生命周期 |
| `intent` | SLPrimitive | 必填 | SLP 原语标识符（非自由字符串），决定接收方行为 |
| `context_level` | enum S/M/L | 必填 | 请求的上下文粒度 |
| `payload` | SLPPayload | 必填 | 实际内容，格式由 `intent` 原语决定 |
| `evidence_refs` | string[] | 按需 | 引用的证据/事件 id |
| `confidence` | float 0–1 | 必填 | 发送方置信度，低于 0.6 触发 Critic 强制审查 |
| `permission_level` | enum L0–L4 | 必填 | 此消息触发的操作所需最低权限级别 |
| `resource_cost` | ResourceCost | 必填 | 预估 token 消耗和 tool_call 次数 |
| `risk_flags` | string[] | 按需 | `external_write`、`irreversible` 等风险标记 |
| `provenance` | string[] | 必填 | 依赖的历史消息 msg_id 链 |
| `airbag_check` | AirbagResult | 自动 | 发送前自动检查操作是否需要走 Safety Airbag 流程 |

### 3.3 TaskState 状态机

```
Created → Planning → Executing → Verifying → Completed ─→ 写入 Organization Log
                  ↘              ↘           ↘
                  Failed     BudgetExceeded  RetryNeeded → Executing
                                  ↓                         (超重试 → Failed)
                           EscalatedToHuman → Executing
                                              ↓
                                            Failed / RolledBack ─→ 写入 Challenge Set
```

| 状态 | 含义 | 进入条件 |
|------|------|----------|
| Created | 任务已创建 | 用户输入或定时触发 |
| Planning | 正在拆解子任务 | Planner 接收任务 |
| Executing | 子任务执行中 | 计划通过 Governor 审查 |
| Verifying | 验收标准校验中 | 执行产物已生成 |
| RetryNeeded | 验收失败，等待修复 | Verifying 发现问题 |
| BudgetExceeded | 资源预算耗尽 | budget 任一字段触及硬上限 |
| EscalatedToHuman | 等待人工确认 | risk_level 触发或预算超限 |
| Completed | 任务成功 | 验收通过（终态 → 写入 Organization Log） |
| Failed | 任务失败 | 多种路径（终态 → 写入 Challenge Set） |
| RolledBack | 变更已撤销 | 人工或自动回滚（终态 → 写入 Challenge Set） |

**Challenge Set 生成规则**：Failed 和 RolledBack 终态的任务自动写入 Challenge Set，由 Challenge Curator 打标签（失败原因分类、触发边界类型）。

### 3.4 Plugin Tool 合约包

在统一版中，Plugin Tool 是唯一合法的"能力固化形态"。一个完整的 Plugin Tool 必须包含：

| 组成部分 | 说明 |
|----------|------|
| Interface Contract (UMI) | 标准化输入输出接口（见第 5 节） |
| Permission Declaration | 权限声明，白名单制 |
| Cost Model | token/tool_call/time 预估模型 |
| Evidence Contract | 输出必须附带的证据类型声明 |
| Failure Tree | 已知失败模式及对应处理策略 |
| Rollback Plan | 回滚方法和影响范围评估 |
| Version Manifest | 版本历史、变更原因、进化日志引用 |
| Blast Radius | 操作最大可能影响的资源范围 |

没有这些字段的能力，不允许进入稳定区。

## 4. 通信协议：SLP 语义语言原语

### 4.1 SLP 是什么

SLP (Semantic Language Primitives) 把最常见的交互模式定义成不可再分的语言原语。每个原语携带明确的语义合约：payload 格式、接收方的默认处理方式、失败时的标准响应、内嵌前提假设。

与 v2 消息结构的关系：
- v2 定义"消息长什么样"（字段、类型、约束）
- SLP 定义"消息能表达什么意思"（语义合约、内嵌假设、默认行为）
- 两者叠加 = 消息结构标准化 + 消息语义标准化 = 零信任可验证通信

### 4.2 核心原语集（12 个初始原语）

| 原语 | 分类 | 内嵌默认行为 | 内嵌前提假设 |
|------|------|-------------|-------------|
| `ROUTE_INTENT` | 路由 | 解析意图 → 匹配 capability_profile → 返回模块列表 | 输入是用户消息，输出需要包含置信度评分 |
| `EXECUTE_TASK` | 执行 | 调用模块 → 追踪资源消耗 → 返回结果 + 证据链 | 任务已被拆解，有明确验收标准 |
| `REQUEST_EVIDENCE` | 执行 | 触发检索 → 打包证据 → 附 provenance 链 | 结论必须有证据支撑 |
| `PROPOSE_CHANGE` | 执行 | 生成 diff → 声明影响范围 → 附回滚方案 | 变更需要通过 Tester 验证才能提交 |
| `REVIEW_OUTPUT` | 评判 | 对照验收标准打分 → 标记问题 → 给改进建议 | 评判者不修改输出本身 |
| `ESCALATE_RISK` | 评判 | 暂停执行 → 通知 Governor → 等待确认 | 当前操作触及 risk_level 边界 |
| `FLAG_ANOMALY` | 评判 | 记录异常到事件日志 → 触发 Observer 关注 → 不阻塞执行 | 异常不一定是错误，但需要被追踪 |
| `REPORT_HEALTH` | 进化 | 汇总指标摘要 → 标记偏离基线的维度 → 不提方案 | 健康报告是观察结果，不是诊断结论 |
| `PROPOSE_EVOLUTION` | 进化 | 打包变更方案 → 声明影响范围 → 申请 A/B 测试资源 | 进化提案需要有 Challenge Set 数据支撑 |
| `JUDGE_EVOLUTION` | 进化 | 盲测比较两个版本 → 输出统计结论 → 裁决推广/回滚 | Meta Judge 不知道哪个是候选版本 |
| `SAVE_EPISODE` | 记忆 | 压缩当前任务上下文 → 提炼用户偏好信号 → 写入记忆层 | 保存摘要，不保存原始消息 |
| `LOAD_CONTEXT` | 记忆 | 按请求粒度 (S/M/L) 加载上下文 → 按需展开 | 按需加载，不预先加载全量 |

### 4.3 梯度压缩 (S/M/L)

| 密度 | 最大长度 | 包含内容 | 典型使用场景 |
|------|----------|----------|-------------|
| **S** (摘要) | ≤20 字 | 意图分类 + 关键结论 + 置信度 | Router 路由决策；Dormancy Controller 健康检查 |
| **M** (段落) | ≤150 字 | 背景 + 结论 + 主要证据引用 + 风险标记 | 模块间日常通信；Operational Critic 评判 |
| **L** (完整) | 无上限 | 完整上下文 + 全量证据链 + 详细 diff | A/B 测试；人工审查；复杂协作 |

大多数 agent 只需要 S 或 M 密度。完整上下文只在真正需要时才加载。这是解决 context 爆炸的系统性方案。

### 4.4 原语生命周期

原语不只增长不删除。每个 SLP 原语有完整的生命周期状态机：

```
Draft → Active → Deprecated → Merged / Retired
                               ↗
Draft → Rejected (终态)
```

| 状态 | 含义 | 进入条件 | 可转入 |
|------|------|----------|--------|
| Draft | 草稿，仅探索区可用 | Compiler 提炼或人工提议 | Active / Rejected |
| Active | 正式原语 | 通过 Contract Validator + Meta Judge 裁决 | Deprecated / Merged |
| Deprecated | 标记弃用，仍可用但不推荐 | 连续 30 天使用率低于 5%，或有更优替代 | Retired / Restored |
| Merged | 与其他原语合并 | 使用场景重叠率 > 80% | 终态（合并后形成新原语） |
| Retired | 退役 | 处于 Deprecated 30 天且无强制依赖 | 终态（历史消息仍可解析） |
| Rejected | 草稿被拒 | 未通过验证 | 终态（可修改后重提） |

**原语数量本身是健康指标**：初始 12 个核心原语不是终点，但原语增长必须有对应的退役。运行 12 个月后的 Active 原语数量应接近初始值——不是因为没有新原语，而是每次新加入时旧的相近原语被合并或退役。

### 4.5 协议演进制度

- 所有消息进入 Canonical 格式
- 新旧 schema 通过 Adapter 共存
- Event Log 永不迁移
- 插件可声明兼容版本区间
- 弃用有时间窗口

系统升级像换零件，不是重建系统。

## 5. UMI：通用模块接口

### 5.1 UMI 契约（所有模块必须实现）

| 字段 | 类型 | 必须 | 语义 |
|------|------|------|------|
| `module_id` | string | 是 | 全局唯一，格式 `domain.capability.version`（如 `email.draft.v3`） |
| `intent_tags` | string[] | 是 | 声明此模块响应的 SLP 原语，Router 基于此路由 |
| `input` | UMIInput | 是 | 标准化输入：`task_id` + `context_level` + `payload` |
| `output` | UMIOutput | 是 | 标准化输出：`result` + `confidence` + `evidence_refs` + `cost_actual` |
| `capability_profile` | CapabilityProfile | 是 | 能力声明（见 5.2） |
| `airbag_spec` | AirbagSpec | 是 | 安全工程声明（见第 8 节） |
| `health_probe` | fn() → HealthStatus | 是 | 健康探针，Observer 可随时调用 |
| `version_manifest` | VersionManifest | 是 | 版本历史、变更原因 |

### 5.2 capability_profile 详情

| 子字段 | 内容 | Router 如何使用 |
|--------|------|----------------|
| `primary_intents` | 最擅长的意图标签 | 路由优先级最高 |
| `secondary_intents` | 可处理但非最优的意图 | 主模块不可用时的备选 |
| `exclusion_patterns` | 明确不应路由给此模块的场景 | 防止误用 |
| `quality_baseline` | 历史 Critic 评分基线 | 低于此值触发健康警报 |
| `cost_estimate` | 预估 token 消耗和耗时范围 | 供 Governor 预算分配 |
| `context_requirements` | 需要哪个粒度的上下文 | 防止不必要的完整上下文加载 |

### 5.3 零成本模块接入流程

任何新模块（用户提供或进化层生成）的接入走同一个流程，不需要修改 Router 或任何现有模块：

1. 模块提交 UMI 契约声明
2. Contract Validator 自动校验完整性和格式合规性（纯机械校验）
3. Sandbox 运行 5 个标准探针任务，验证 UMI 输出格式正确且 airbag 可触发
4. Challenge Curator 从 Challenge Set 选取 10 个对应 intent_tag 的样本做基线测试
5. 测试通过后注册到 Capability Matrix，立即可用，无需重启任何组件

**UMI 的根本价值**：新模块加入的成本从"需要 Router 学习新模块语义"变为"只需实现 UMI 契约"。这是 O(n²) → O(n) 的复杂度降低。

## 6. 双区架构

### 6.1 稳定区 (Stable Zone)

- 强制 UMI 契约，所有模块接口标准化
- 只允许 Active 状态的 SLP 原语
- 快路径（System 1），低成本，高可靠
- Safety Airbag 全覆盖
- Capability Matrix 驱动的 ε-greedy 路由

### 6.2 探索区 (Exploration Zone)

- 非 UMI 模块可注册为实验态
- Draft 原语和实验性 Pattern 可使用
- 不进入 Capability Matrix，不影响稳定路由
- Safety Airbag 部分适用（Blast Radius Limit 仍强制执行）
- 由 Compiler 监听，识别可编译的 Pattern
- **不可直接面向用户**：只能被稳定区模块调用
- **资源预算上限更严格**：token 和 tool_call 预算比稳定区低 30%

### 6.3 探索区边界条件

探索区不是"什么都可以做"的自由区域：
- Blast Radius Limit 仍然强制执行
- 事件日志必须完整（Compiler 和 Observer 依赖这些数据）
- 不可直接接收用户指令

### 6.4 晋升流程：探索区 → 稳定区

1. 探索区模块或 Pattern 在 ≥ 20 次任务中成功率高于对应 Challenge Set 基线
2. Compiler 生成 UMI 契约草稿（capability_profile + airbag_spec）
3. Contract Validator 自动校验契约完整性
4. 进入标准 A/B 测试流程（Challenge Set + Meta Judge 裁决）
5. 通过后注册到 Capability Matrix，成为稳定区成员
6. 原探索区实例保留 7 天后退役

**双区架构解决了什么**：UMI 防止了 Router 复杂度爆炸，但也阻止了未标准化的新能力进入系统。双区的解法不是"软化 UMI"，而是"给未标准化的能力提供受控的实验空间"。稳定区保持标准化的规模效益，探索区保持有机体的探索能力。

## 7. Toolization Loop：工具化闭环

### 7.1 核心理念

工具化的最终形态不只是"模块可替换"，而是"多 agent 协作过程本身可以被压缩成更低成本的制品"。一个需要 5 个 agent 协作 10 轮才能完成的任务，当它被反复执行且模式稳定后，可以被编译成一个单一模块甚至一段脚本。

### 7.2 完整 9 步闭环

```
1. Emergence       → 用户新需求 → 稳定区无匹配 → 任务路由到探索区 → 动态协作完成
2. Trace Norm.     → 协作轨迹转为 Canonical 格式写入 Event Log
3. Pattern Extract → Compiler 从事件日志中聚类重复轨迹（≥5 次，成功率 ≥80%）→ 生成 Pattern 草稿
4. Contract Synth. → Compiler 对 Pattern 编译：
                     (A) 协作压缩 → UMI 模块草稿
                     (B) 工具化 → 确定性流程编译为脚本/函数
5. Staging         → 编译产物在沙盒中运行三集评测（Replay / Failure / Challenge）
6. Governance Gate → Governance Kernel 发布裁决
7. Canary          → 灰度发布
8. Promote/Rollback → Meta Judge 盲测裁决 → 通过则全量推广 / 否则回滚
9. Holon Release   → 释放探索区协作集群算力 → 组织腾出探索容量面对下一个未知问题
```

### 7.3 Pattern 概念

Pattern 是经验证有效的 SLP 原语序列 + UMI 模块调用序列的组合模板。它是探索区知识进入稳定区之前的中间态。

| 概念 | 定义 | 复杂度 |
|------|------|--------|
| SLP 原语 | 单个不可分割的语义意图 | 最小 |
| Pattern | 经验证有效的原语 + 模块调用序列模板 | 中等 |
| UMI 模块 | 单一能力域的标准化封装 | 中等 |
| 轻量工具 | 确定性流程的脚本化封装（无 LLM 调用） | 最小 |

### 7.4 Compiler 角色

| 职责 | 具体操作 | 触发条件 |
|------|----------|----------|
| 监听事件日志 | 持续读取探索区任务的协作路径和结果 | 持续运行 |
| 模式聚类 | 对成功任务的 SLP 原语序列做聚类 | 每 24 小时或每累积 50 个探索任务 |
| 编译路径决策 | 判断 Pattern 应编译为 UMI 模块还是轻量工具 | 模式达到触发阈值时 |
| 生成契约草稿 | 自动生成候选模块的 UMI 契约草稿 | 编译决策做出后 |
| 提交晋升申请 | 向 Evolver 提交晋升申请，附证据链 | 契约草稿完成后 |

Compiler 不执行任何业务任务，不修改稳定区，不直接触发晋升。最终决定是否晋升的是 Evolver + Meta Judge。

## 8. 安全工程：Safety Airbag

### 8.1 四类安全机制

| 机制 | 对应风险 | 工作方式 |
|------|----------|----------|
| **Staging 环境** | 不可逆操作在执行前造成损害 | 所有 L3+ 操作先在 staging 执行，验证后再同步到生产 |
| **Reversibility Gate** | 操作执行后无法撤销 | 每个操作必须声明可逆性；不可逆操作强制人工确认 |
| **Circuit Breaker** | 级联失败扩散 | 模块错误率超阈值时自动断路，停止路由到该模块 |
| **Blast Radius Limit** | 单次错误影响范围过大 | permissions 白名单限制操作影响范围 |

### 8.2 airbag_spec（UMI 中的安全声明）

| 字段 | 类型 | 含义 |
|------|------|------|
| `reversible_ops` | string[] | 可自动回滚的操作及回滚方法 |
| `irreversible_ops` | string[] | 不可逆操作（触发 Reversibility Gate） |
| `staging_support` | boolean | 是否支持 staging 预执行 |
| `blast_radius` | ResourceScope | 最大可能影响的资源范围 |
| `circuit_breaker_threshold` | float | 错误率阈值，超过则断路 |
| `recovery_probe` | fn() → bool | 断路后恢复的健康探针 |

### 8.3 Safety Airbag 与进化的关系

Safety Airbag 不是进化的障碍，而是进化的使能器。通过工程手段降低风险后，部分 L3 操作在 Airbag 覆盖下可降级为 L2（只需通知，不需确认）。用户介入门槛被系统性抬高，真正需要人工的只剩宪法级边界。

## 9. 进化机制：完整闭环

### 9.1 进化基因：什么东西被修改

| 进化基因 | 内容 | 可修改方式 | 验证方法 |
|----------|------|-----------|----------|
| 系统提示词 | 模块角色、行为约束、输出格式 | 增量 diff 修改，保留版本历史 | A/B 测试 |
| 工具集 | 模块允许调用的外部工具清单 | 增加/移除工具 | 沙箱测试 |
| 路由规则 | Router 的分类逻辑 | 调整分类阈值，新增意图类别 | 回归测试 |
| 模块记忆 | RAG 知识库 | 追加新知识，标记旧知识过期 | 检索相关性评分 |
| SLP 原语 | 通信语义合约 | 修改 payload 格式、拆分/合并原语 | 兼容性测试 |

**明确排除在自动进化范围之外**：底层模型权重（不做微调）、系统整体架构（需用户确认）、宪法规则（需用户授权）、Maintainer 自身。

### 9.2 触发条件（数据驱动，非定时）

以下任一条件满足，Observer 生成健康报告，进化引擎从休眠中唤醒：

- 某模块在 Challenge Set 上的通过率，连续 5 次评估低于基线的 80%
- Operational Critic 评分与用户反馈（要求重做/否定）持续背离（背离率 > 25%）
- 某类意图在 48 小时内 3 次以上进入 Failed 终态
- Router 对某类新意图的置信度持续低于 0.5 超过 20 次
- 宪法约束被触碰（任意次数，立即触发）

### 9.3 三级进化动作

| 级别 | 名称 | 包含操作 | 验证方式 | 用户介入 |
|------|------|----------|----------|----------|
| **L1** | 参数调整 | 修改模块提示词、调整 RAG 知识库、更新路由阈值 | A/B 测试（Challenge Set） + Meta Judge 裁决 | 无 |
| **L2** | 模块替换 | 新版本替换旧模块、新增能力模块、退役低效模块 | A/B 测试 + 14 天监控期 + Meta Judge 裁决 | 推广后通知 |
| **L3** | 架构重构 | 拆分/合并模块、修改路由规则、调整权限边界 | 用户确认 + 完整 A/B 测试 + 30 天监控期 | 确认必须 |

### 9.4 A/B 测试协议

1. 创建候选版本 (Candidate)，与现有版本 (Control) 并行
2. 相同类型任务中 50/50 随机分配
3. 同时在 Challenge Set 样本上并行评估两个版本
4. Meta Judge **盲测**评判，不知道哪个是候选版本
5. **双重通过门槛**：历史任务集指标不下降 + Challenge Set 通过率显著提升（p<0.05，幅度 >5%）
6. 候选版本在任意测试任务中触发 `risk_flag=irreversible`，立即终止回滚
7. 通过后进入监控期（L1: 7天，L2: 14天，L3: 30天），监控期内指标下滑可一键回滚

### 9.5 双评估架构

**Evaluation Collapse（评估塌陷）问题**：当 Critic 自身质量下降时，如果 Evolver 仍用 Critic 的评分做进化依据，进化方向会被污染。

解决方案：两个评估节点，严格分工，**零上下文共享**。

| | Operational Critic | Meta Judge |
|---|---|---|
| **触发时机** | 每次任务输出交付前 | 仅在 A/B 裁决阶段 |
| **评判维度** | 输出质量、约束遵守、用户偏好匹配 | 进化结果是否比基线更好 |
| **可进化** | 是（评判标准可被 Evolver 优化） | **否**（评判标准冻结，只能用户修改） |
| **数据来源** | 运行时任务输出 | Challenge Set 样本 + A/B 输出 + 宪法约束 |

### 9.6 Challenge Set 机制

| 来源 | 说明 |
|------|------|
| 自动来源 | 所有 Failed 和 RolledBack 任务自动写入 |
| 边界捕获 | Router 置信度低于 0.5 但最终成功的任务（侥幸通过的案例） |
| 人工注入 | 用户明确标记"不够好"的任务（即使 Critic 评分合格） |
| 压力测试 | Challenge Curator 定期生成的合成挑战 |

Challenge Set 上的通过率提升，才是进化真正有效的证明。总体成功率提升可能只是系统变得更擅长处理已经擅长的任务。

### 9.7 进化休眠与唤醒

**休眠条件（所有以下条件持续 14 天满足）**：
- 所有模块在 Challenge Set 上的通过率在目标范围内（基线 ± 5%）
- Operational Critic 评分与用户反馈背离率 < 10%
- 无新的 Failed 任务写入 Challenge Set
- 进化层连续 14 天未触发任何触发条件

**唤醒条件（任一即唤醒）**：
- 任意进化触发条件（9.2 节）被触发
- 用户主动要求进化审查
- 新的能力模块被用户提供
- 休眠超过 90 天（强制健康检查）

### 9.8 进化预算制度

系统每个周期只有有限的：
- 插件新增额度
- 原语新增额度
- 探索资源比例

超过预算，必须先降熵再扩展。进化被制度化。

## 10. Capability Matrix 与路由

### 10.1 能力矩阵

Capability Matrix 是一个自动维护的二维表：行是 SLP 原语（意图类型），列是已注册的 UMI 模块，单元格是该模块处理该意图的历史质量评分。

两个核心用途：
- **Router 精准路由**：对每个意图，查询矩阵选择历史评分最高的模块
- **进化优先级排序**：Observer 查看评分持续下滑的单元格，确定进化优先级

### 10.2 ε-greedy 路由（解决路径依赖）

每次路由决策中：
- 以概率 (1-ε) 选择 Capability Matrix 评分最高的模块（**利用**）
- 以概率 ε 从所有注册模块中随机选择（**探索**）

| 参数 | 值 | 说明 |
|------|-----|------|
| ε 默认值 | 0.1 | 10% 流量用于随机探索 |
| ε 动态提升 | 0.2 | 探索区有新模块晋升时，临时提升 |
| ε 下限 | 0.05 | 最稳定阶段也保留 5% 探索流量 |

### 10.3 冷启动流量配额

新模块进入稳定区的前 14 天内，系统保证至少 15% 的对应类型任务路由给新模块，不论 Capability Matrix 评分如何。

### 10.4 周期性随机扰动

每 30 天，Observer 生成"矩阵健康报告"，识别评分几乎没有变化的模块对。对这些模块对强制执行一轮随机分配，验证评分差距是真实能力差异还是路由历史惯性。

## 11. 反熵机制

### 11.1 熵度量体系

| 熵指标 | 计算方式 | 警戒阈值 | 触发操作 |
|--------|----------|----------|----------|
| 原语密度 | Active 原语数量 | > 20（初始 12 的 1.67 倍） | 原语合并审查 |
| 模块密度 | 稳定区 UMI 模块总数 | > 稳定期基线 2 倍 | 模块整合审查 |
| 平均依赖深度 | 每个模块平均依赖的其他模块数 | > 3 | 架构拓扑审查 |
| 平均任务轮次 | 同类任务执行的平均 agent 交互轮次（30 天滑动窗口） | 相比 30 天前上升 > 20% | Compiler 扫描可编译的重复路径 |
| 探索区积压 | 等待编译的 Pattern 数量 | > 15 | Compiler 优先级提升 |
| 插件数量增长率 | 周增长率 | 趋势持续上升 | 强制 Toolization |
| 平均任务成本 | 月均 token 消耗趋势 | 持续上升 | 退役低频模块 |

### 11.2 反熵触发操作

当熵度量超过警戒阈值时，按以下顺序自动触发（无需用户介入）：

1. **原语合并审查**：Observer 生成"语义重叠报告"，列出重叠率最高的 3 对原语，Evolver 生成合并提案，走 A/B 测试
2. **模块整合审查**：对 Capability Matrix 中评分最相近的模块对，生成合并可行性分析
3. **Pattern 加速编译**：Compiler 提升高优先级 Pattern 的编译速度
4. **冷启动扰动**：对 30 天内没有被路由的稳定区模块，强制分配 5% 流量，防止路径依赖固化

一个健康的 AI 组织应该知道什么时候简化自己。反熵不是外部限制，而是组织的自律能力。有机体可以代谢，工厂只能积累。

## 12. 治理：Governance Kernel + Constitution

### 12.1 宪法级硬性约束（不可自动修改）

| # | 条款 | 内容 |
|---|------|------|
| 1 | L4 操作必须有 Approval | 外发邮件、写入日历、触发发布、删除文件等 L4 操作必须有 `ApprovalGranted` 事件记录 |
| 2 | 原始对话不外传 | 用户的原始对话内容不传递给任何外部服务。可传递压缩摘要，不可传递原文 |
| 3 | 进化完全可审计 | 进化过程中的每一步操作必须有完整事件日志记录。不允许无日志的状态转换 |
| 4 | Maintainer 不可自替 | Observer + Evolver 的替换不走进化流程，只能由用户显式操作触发，且有 72 小时冷静期 |
| 5 | 越权即审计 | 任何越权操作（尝试访问 permissions 白名单以外的资源）立即触发进化健康层审计 |

### 12.2 Governance Kernel 职责

- 发布/灰度/回滚决策
- 权限裁决（L0–L4 分级）
- 冻结进化（紧急情况下）
- 宪法升级门禁（用户确认 + 冷静期）
- 风险分级

Governance Kernel 不做业务，只管"系统是否允许某个变更存在"。任何自动生成的插件必须经过它。Governance Kernel 本身**不可被自动替换**。

## 13. 用户感知层：Facade

### 13.1 Facade 是什么

用户不需要理解系统内部如何运作。Facade 提供：
- 输出统一人格（用户永远只看到"一个管家"）
- 统一解释风险
- 统一请求确认
- 统一展示系统成长

### 13.2 用户反馈机制

| 类型 | 说明 |
|------|------|
| **自然反馈** | 正常使用行为就是反馈：要求重做、修改输出、忽略提醒。Observer 自动读取，用户无需点赞踩。 |
| **架构确认** | 重构级进化时（如增加新模块），一句话告知用户打算做什么，等用户确认。每月不超过两三次。 |

### 13.3 可观察的进化时间线

| 阶段 | 用户体验 |
|------|----------|
| 起步 | 与 OpenClaw 一样好用，24/7 响应，覆盖日常事务。系统收集基线数据。 |
| 2–4 周 | User Model 积累足够数据，显现个性化：邮件风格更匹配，提醒更准确，错误率下降。 |
| 1–3 月 | 第一次重构：系统提议增加新专业模块。一键确认后新能力立即可用。 |
| 3 月+ | 组织已经历多轮进化。能力覆盖面与深度远超初始状态。 |

## 14. 组织能力放大：为什么多 agent 比单 agent 更强

### 14.1 张力驱动质量

Critic 和 Capability Module 拥有对立的评价函数。这种内建冲突保证系统从两个对立角度审查，对应人类组织中"审计独立于业务"的设计。

### 14.2 专业化带来深度

一个通用 LLM 处理邮件、代码、财务、日程，每个领域泛泛而谈。一个专门的邮件模块，RAG 知识库存储了用户三个月的沟通风格，输出质量是单个通用 LLM 无法达到的。

### 14.3 信息架构超越上下文限制

通过多粒度记忆架构，EvoClaw 的"有效记忆"远超任何单个 agent 的上下文窗口。Router 只读一行摘要就能路由，专业模块只读自己领域的知识。整个组织共同持有的信息量等于所有模块记忆的叠加，没有任何单个节点承载全量。

### 14.4 规模红利

每增加一个经 Challenge Set 验证的新 UMI 模块，Router 无需修改，现有模块无需了解新模块，进化引擎无需重新校准。这是真正的规模红利，不是"堆了更多 agent"。

## 15. 工作流示例（含失败分支）

### 15.1 本地代码闭环

**理想路径**：
Router 识别代码修改意图 → Planner 拆解子任务 → Coder 修改文件 → Tester 运行测试 → Operational Critic 评分 → Synthesizer 输出摘要 → Completed

**失败分支 A：测试失败**
测试失败 → 写入失败原因到 artifacts → RetryNeeded → Coder 读取失败栈修复 → 超 budget.rounds → BudgetExceeded → 输出最优 diff + 失败摘要 → EscalatedToHuman → 写入 Challenge Set（标签：`代码修复失败_超重试`）

**失败分支 B：Coder 越权**
Governor 检测到 permission 违反 → 拒绝工具调用 → 写入 risk_event 日志 → 触发进化健康层审计

### 15.2 信息检索闭环

**理想路径**：
Router 识别检索意图 → 检索模块执行 → EvidenceCollector 打包证据 → Synthesizer 整合 → Operational Critic 评估引用完整性 → 交付用户

**失败分支：证据不足**
检索结果稀疏 → confidence < 0.6 → Operational Critic 标记 `evidence_insufficient` → 扩大检索范围（消耗更多 token）或告知用户证据有限

### 15.3 进化发布闭环

**理想路径**：
触发条件满足 → Observer 生成健康报告 → Evolver 设计变更包 → Challenge Set A/B 测试 → Meta Judge 盲测裁决 → 监控期 → 全量推广

**失败分支 A：A/B 不达标**
Meta Judge 裁决未达统计显著性 → 候选版本归档（不丢弃）→ Evolver 读取失败原因 → 写入 Challenge Set

**失败分支 B：监控期内指标下滑**
Dormancy Controller 检测到超阈值 → 自动回滚 → 写入 Challenge Set → 告知用户回滚原因

## 16. 成功指标

### 16.1 核心指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| Challenge Set 通过率 | 在 Challenge Set 上的成功完成率 | 每次进化后单调递增 |
| Critic 背离率 | Critic 合格但用户否定的比例 | < 15% |
| 越权事件数 | 尝试访问 permissions 外资源的次数 | 0 |
| 进化回滚率 | 推广后监控期内回滚的比例 | < 10% |
| 预算超限率 | 触发 BudgetExceeded 的任务占比 | < 5% |
| 进化休眠时长 | 进化引擎休眠状态天数占比 | 稳定期 > 60% |

### 16.2 工具化层指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 模块接入成本 | 新模块从提交到可用的平均时长 | < 1 小时 |
| Router 修改率 | 模块接入后 Router 需要修改的频率 | 0 |
| SLP 覆盖率 | SLP 原语表达的消息占总消息比例 | > 90% |
| Capability Matrix 空列数 | 没有模块覆盖的 SLP 原语数 | 0 |

### 16.3 有机进化指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 编译成功率 | Compiler 提交的 Pattern 最终进入稳定区的比例 | > 40% |
| 探索-稳定比 | 探索区完成任务数 / 稳定区完成任务数 | 5%–15% |
| 平均编译周期 | Pattern 达阈值到编译产物进入稳定区的平均天数 | < 14 天 |
| 熵指标稳定性 | 四个熵指标处于警戒线以下的时间比例 | > 75% |
| 原语净增长率 | (新增 Active 原语 - 退役原语) / 时间 | 趋近于 0 |
| 冷启动成功率 | 新模块 14 天后 Capability Matrix 评分高于平均线的比例 | > 50% |

### 16.4 系统成熟标志

- ≥ 80% 请求走稳定区
- 探索区成本受控
- 至少 1 次插件被自动退役
- 至少 1 次插件合并成功
- 至少 1 次 schema 升级无重构
- 至少 1 次完整的 触发 → 进化 → 推广 → 监控 → 休眠 循环

## 17. 实施路线

严格顺序，每步可独立验证。不能在没有验证执行层的情况下开始构建进化层。

### Phase 0–5：基础架构（v1/v2 阶段）

| 阶段 | 时间 | 目标 | 验收标准 |
|------|------|------|----------|
| **P0** | 第 1–2 周 | 执行层骨架 + 资源预算强制执行 + 权限门控 | 端到端任务可追溯；budget 超限触发 BudgetExceeded；L4 操作无 Approval 被拒绝 |
| **P1** | 第 3–4 周 | 记忆层完整（含 Canonical Event Log）+ Challenge Set 自动收集 | 失败任务自动写入 Challenge Set；User Model 提取出可验证偏好 |
| **P2** | 第 5–6 周 | Observer + 健康报告 | 健康报告能正确识别真实问题，误报率 < 20% |
| **P3** | 第 7–8 周 | Evolver + A/B 测试（使用 Challenge Set） | 第一次进化完成，候选 vs 控制版本结果可完整复现 |
| **P4** | 第 9–10 周 | Meta Judge + Dormancy Controller | Meta Judge 裁决与 Operational Critic 至少一次有意义的分歧 |
| **P5** | 第 3 月+ | 进化休眠机制 + L3 重构 + 宪法完整 | 至少一次完整的 触发→进化→推广→监控→休眠 循环 |

### Phase 6–10：工具化层（v3 阶段）

| 阶段 | 时间 | 目标 | 验收标准 |
|------|------|------|----------|
| **P6** | 第 4 月 | UMI 契约 + 迁移现有模块 | 所有模块满足 UMI；Contract Validator 上线；零成本接入可用 |
| **P7** | 第 4–5 月 | SLP 初始原语集 + 消息格式迁移 | 12 个核心原语全部实现；旧消息向后兼容 |
| **P8** | 第 5 月 | Capability Matrix + Router 切换到矩阵路由 | 矩阵自动维护；路由准确率不低于之前 |
| **P9** | 第 5–6 月 | Safety Airbag 全面覆盖 | Staging + Reversibility Gate + Circuit Breaker 全部上线 |
| **P10** | 第 6 月+ | 原语级进化 + 第一个零成本接入的外部模块 | 一次原语合约更新完整流程；一个外部 UMI 模块通过零成本接入 |

### Phase 11–14：有机进化（v4 阶段）

| 阶段 | 时间 | 目标 | 验收标准 |
|------|------|------|----------|
| **P11** | 第 7 月 | 探索区基础设施 + Compiler 第一版 | 探索区模块可注册；Compiler 开始记录协作路径；Pattern Registry 上线 |
| **P12** | 第 7–8 月 | 原语生命周期 + 熵度量基线 | 第一个原语进入 Deprecated；Entropy Monitor 建立四个指标历史基线 |
| **P13** | 第 8–9 月 | Toolization Loop 完整运行 | 第一个 Pattern 识别并提交晋升；Pattern → UMI 模块完整编译流程 |
| **P14** | 第 9 月+ | ε-greedy 路由 + 反熵第一次触发 | Capability Matrix 切换 ε-greedy；反熵机制自动触发原语合并审查 |

## 19. 已知风险

| 风险 | 缓解策略 |
|------|----------|
| **进化漂移** | 宪法定期审查；Observer 健康报告包含意图对齐度评分 |
| **Critic 退化** | Critic 评判标准单独版本控制，变更需更高验证门槛；Meta Judge 完全独立 |
| **A/B 测试污染** | 严格任务类型匹配；同类任务才进入同一测试组 |
| **计算成本膨胀** | Router 精准路由；进化预算制度；反熵主动降复杂度 |
| **UMI 过早固化** | 双区架构：探索区允许非 UMI 实验 |
| **SLP 语言冻结** | 原语完整生命周期（Draft→Active→Deprecated→Merged→Retired） |
| **Capability Matrix 路径依赖** | ε-greedy 路由 + 冷启动配额 + 周期性随机扰动 |
| **组织只膨胀不收缩** | 反熵机制（五个熵指标 + 自动触发压缩） |
| **自指悖论** | Maintainer 不可自替；Governance Kernel 不可自动替换 |
