0. v3.0 的核心命题：工具化
v1.0 解决了"组织能不能用"，v2.0 解决了"进化能不能信"。v3.0 要解决的问题是：组织能不能无限扩展，同时不让复杂度随规模线性增长。
答案来自观察人类社会的一个普遍模式：工具化。

人类案例	工具化之前	工具化之后
牛顿力学	需要博士级数学推导，才能预测行星轨迹	F=ma 三字，初中生直接使用结果，前提假设已内嵌于公式
汽车驾驶	司机须了解发动机、传动、点火系统全部原理	标准化接口（方向盘/油门/刹车）+ 安全工程，普通人可驾驶
智能手机	需要硬件工程师才能操作计算机	App 标准化接口，任何人可以使用任何 App，无需懂底层

这三个案例背后是同一个机制，三个要素缺一不可：
•复杂度封装：内部实现对调用者透明，只暴露接口
•接口标准化：同类事物的交互方式完全一致，"会一个等于会所有"
•失败代价工程化降低：安全气囊、ABS、App 审核机制，让普通人犯错的代价可承受

v3.0 的核心主张
把工具化的三个要素直接应用于 AI 组织：能力模块的接口标准化（UMI）+ 通信语言的复杂度封装（SLP）+ 高风险操作的安全工程（Safety Airbag）。这三者组合，才能让组织的规模扩展与复杂度增长解耦。
0.1 工具化解决的根本矛盾
v1.0 和 v2.0 的设计都隐含了一个假设：组织里的每个 agent 需要"理解"任务上下文，才能正确协作。这个假设在组织规模小的时候成立，但随着模块数量增加，它会产生一个根本矛盾：
每新增一个模块，Router 需要理解这个模块的能力边界；每个模块需要理解组织的协作规范；每次进化需要验证所有模块的行为是否符合预期。这是一个 O(n²) 的复杂度增长曲线。
工具化的目标是把这个曲线压回 O(n)：新模块只要满足标准接口就能接入，不需要"学习"组织文化；Router 不需要理解每个模块的内部实现，只看接口声明；进化验证只需要验证接口契约是否仍然满足。

2. UMI：通用模块接口（Universal Module Interface）
这是工具化层的第一个支柱，对应汽车案例里的"标准化方向盘"。
v1.0 和 v2.0 的模块接口是自定义的——每个模块声明自己的 inputs/outputs 格式，Router 需要理解每个模块的特定语义才能正确调用。这在模块少时没问题，但随着模块增加，Router 的复杂度线性增长。
UMI 的解决方案：所有能力模块，无论其内部多么不同，对外暴露完全相同的调用契约。Router 只需要理解 UMI，就能调用任意模块。
2.1 UMI 契约（所有模块必须实现）

UMI 字段	类型	是否必须	语义
module_id	string	必须	全局唯一标识符，格式：domain.capability.version（如 email.draft.v3）
intent_tags	string[]	必须	声明此模块响应的标准语义原语（见第3节），Router 基于此路由
input	UMIInput	必须	标准化输入：task_id + context_level + payload（格式由 intent_tag 决定）
output	UMIOutput	必须	标准化输出：result + confidence + evidence_refs + cost_actual
capability_profile	CapabilityProfile	必须	能力声明：擅长/不擅长的场景、期望质量基线、适用 risk_level
airbag_spec	AirbagSpec	必须	安全工程声明：哪些操作可逆、回滚方法、staging 支持（见第4节）
health_probe	fn() → HealthStatus	必须	健康探针：Observer 可随时调用，返回模块当前状态和性能摘要
version_manifest	VersionManifest	必须	版本历史：当前版本、上一版本引用、变更原因（进化日志）

UMI 的根本价值
一个新的能力模块加入 EvoClaw 组织的成本，从"需要 Router 学习新模块语义"变为"只需要实现 UMI 契约"。这是 O(n²) → O(n) 的复杂度降低。每一个满足 UMI 的模块，Router 无需任何改动就能立即调用。
2.2 capability_profile：让 Router 做精准路由
capability_profile 是 UMI 里最重要的字段。它不是自由文本描述，而是结构化的能力声明，让 Router 可以用机械规则做路由决策，而不是每次都用 LLM 判断。

capability_profile 子字段	内容	Router 如何使用
primary_intents	string[]	此模块最擅长的意图标签，路由优先级最高
secondary_intents	string[]	可以处理但不是最优的意图，在主模块不可用时的备选
exclusion_patterns	string[]	明确不应该路由给此模块的场景（防止误用）
quality_baseline	QualityMetrics	历史 Critic 评分基线，低于此值触发健康警报
cost_estimate	CostRange	预估 token 消耗和耗时范围，供 Governor 预算分配
context_requirements	ContextSpec	此模块需要哪个粒度的上下文，防止不必要的完整上下文加载
2.3 零成本模块接入流程
任何新能力模块（无论是用户提供还是进化层生成）的接入都走同一个流程，不需要修改 Router 或任何现有模块：
1.模块提交 UMI 契约声明（module_id、intent_tags、capability_profile、airbag_spec）
2.Contract Validator 自动校验声明完整性和格式合规性（纯机械校验，无 LLM 参与）
3.Sandbox 运行 5 个标准探针任务，验证 UMI 输出格式正确且 airbag 可触发
4.Challenge Curator 从挑战集中选取 10 个对应 intent_tag 的样本做能力基线测试
5.测试通过后注册到 Router 的模块注册表，立即可用，无需重启任何组件


3. SLP：语义语言原语（Semantic Language Primitives）
这是工具化层的第二个支柱，对应"牛顿力学被压缩进 F=ma"的案例。
v1.0 和 v2.0 定义了消息的结构（msg_id、intent、confidence 等），但 intent 字段的值是自由字符串，没有固定的语义合约。这意味着每条消息的接收方都需要解析自然语言来理解发送方的意图，噪音高，误解率也高。
SLP 把最常见的交互模式定义成不可再分的语言原语——有点像编程语言的关键字。每个原语携带明确的语义合约：它规定了 payload 的格式、接收方的默认处理方式、失败时的标准响应，以及它内嵌的前提假设。
3.1 原语设计原则
•原子性：每个原语代表一个不可再分的交互意图，不能是多个意图的组合
•完备性：覆盖 EvoClaw 90% 以上的日常交互场景（长尾用自由扩展原语处理）
•内嵌假设：每个原语内嵌一组默认假设，接收方不需要每次协商
•梯度压缩：每个原语支持三个表达密度，发送方按需选择（见 3.3）
3.2 核心原语集（v3.0 初始定义）

原语名称	分类	内嵌默认行为	内嵌前提假设
ROUTE_INTENT	路由	解析意图 → 匹配 capability_profile → 返回模块列表	输入是用户消息，输出需要包含置信度评分
EXECUTE_TASK	执行	调用模块 → 追踪资源消耗 → 返回结果 + 证据链	任务已被 Planner 拆解，有明确验收标准
REQUEST_EVIDENCE	执行	触发检索 → 打包证据 → 附 provenance 链	结论必须有证据支撑，置信度由证据质量决定
PROPOSE_CHANGE	执行	生成 diff → 声明影响范围 → 附回滚方案	变更需要通过 Tester 验证才能提交
REVIEW_OUTPUT	评判	对照验收标准打分 → 标记具体问题 → 给出改进建议	评判者不修改输出本身，只评判
ESCALATE_RISK	评判	暂停当前执行 → 通知 Governor → 等待确认	当前操作触及 risk_level 边界，需要更高权限
FLAG_ANOMALY	评判	记录异常到事件日志 → 触发 Observer 关注 → 不阻塞执行	异常不一定是错误，但需要被追踪
REPORT_HEALTH	进化	汇总指标摘要 → 标记偏离基线的维度 → 不提方案	健康报告是观察结果，不是诊断结论
PROPOSE_EVOLUTION	进化	打包变更方案 → 声明影响范围 → 申请 A/B 测试资源	进化提案需要有挑战集数据支撑
JUDGE_EVOLUTION	进化	盲测比较两个版本 → 输出统计结论 → 裁决推广/回滚	Meta Judge 不知道哪个是候选版本
SAVE_EPISODE	记忆	压缩当前任务上下文 → 提炼用户偏好信号 → 写入记忆层	保存摘要，不保存原始消息
LOAD_CONTEXT	记忆	按请求粒度（S/M/L）加载上下文 → 按需展开	按需加载，不预先加载全量上下文
3.3 梯度压缩：同一原语的三个密度
每个 SLP 原语的 payload 支持三个表达密度，发送方根据场景选择，接收方按 context_requirements 声明需要哪个级别：

密度	最大长度	包含内容	典型使用场景
S（摘要）	20 字	意图分类 + 关键结论 + 置信度	Router 做路由决策；Dormancy Controller 做健康检查；快速过滤
M（段落）	150 字	背景 + 结论 + 主要证据引用 + 风险标记	大多数模块间的日常通信；Operational Critic 评判；Observer 读取
L（完整）	无上限	完整上下文 + 全量证据链 + 详细 diff + 操作历史	进化 A/B 测试；人工审查；复杂任务的模块间协作

这对应"初中教材 vs 研究生教材"——同一个知识，不同的压缩程度，适用不同的使用场景。大多数 agent 只需要 S 或 M 密度，完整上下文只在真正需要时才被加载。这是解决上下文爆炸的系统性方案，而不是临时补丁。
3.4 SLP 与 v2.0 消息结构的关系
v2.0 的 Message Schema 定义了消息的结构骨架。v3.0 的 SLP 把 intent 字段从自由字符串升级为受约束的原语标识符，并给每个原语绑定了 payload 的标准格式。两者关系如下：
•v2.0 定义"消息长什么样"（字段、类型、约束）
•v3.0 SLP 定义"消息能表达什么意思"（语义合约、内嵌假设、默认行为）
•两者叠加：消息结构标准化 + 消息语义标准化 = 真正的零信任可验证通信


4. Safety Airbag：安全工程层
这是工具化层的第三个支柱，对应汽车案例里的"安全气囊和 ABS"。
汽车之所以能让普通人驾驶高速行驶，不只是因为接口简单，更是因为工程系统把错误的代价降低到可承受范围内。ABS 在轮胎打滑时介入，安全气囊在碰撞时展开——这些机制不是因为司机技术好才能触发，而是系统性的保底。
EvoClaw 需要同样的系统性保底，才能让 agent 在更高自主度下安全运行。Safety Airbag 不是靠 agent 的"谨慎"来保证安全，而是靠工程结构。
4.1 三类安全机制

机制类型	对应风险	工作方式	适用场景
Staging 环境	不可逆操作在执行前造成损害	所有 L3+ 操作先在 staging 环境执行，验证结果后再同步到生产	发邮件（draft 预览）·发布代码（staging 部署）·删除文件（软删除）
Reversibility Gate	操作执行后无法撤销	每个操作在执行前必须声明可逆性；不可逆操作强制人工确认	Git push（有对应 revert）·修改日历（保留原始版本）·调用外部 API
Circuit Breaker	级联失败扩散到整个系统	当某模块错误率超过阈值，自动断路（停止路由到该模块）并通知 Observer	模块内部异常·外部依赖不可用·进化推广引入的回归
Blast Radius Limit	单次错误影响范围过大	每次任务的 permissions 白名单限制操作影响范围，不可突破	文件操作只能访问指定路径·网络请求只能访问白名单域名

4.2 airbag_spec：UMI 中的安全声明
每个模块在 UMI 契约中的 airbag_spec 字段必须明确声明以下内容，在模块注册时验证：

字段	类型	含义
reversible_ops	string[]	此模块的哪些操作可以被自动回滚，以及回滚方法
irreversible_ops	string[]	哪些操作是不可逆的（必须触发 Reversibility Gate）
staging_support	boolean	是否支持在 staging 环境预执行
blast_radius	ResourceScope	此模块的操作最大可能影响哪些资源（文件路径、网络域名等）
circuit_breaker_threshold	float	错误率超过此阈值时，Router 应停止路由到此模块
recovery_probe	fn() → bool	断路后模块恢复的健康探针，返回 true 时恢复路由
4.3 安全气囊与进化的关系
Safety Airbag 不是进化的障碍，而是进化的使能器。
v2.0 中，L3 重构级进化需要用户确认，原因是风险太高。v3.0 的 Safety Airbag 通过工程手段降低了这个风险：重构在 staging 环境预执行，影响范围被 Blast Radius Limit 约束，如果出问题 Circuit Breaker 自动断路。
结果是：部分 L3 操作在 Safety Airbag 覆盖下可以降级为 L2（只需通知，不需确认）。用户介入的门槛被系统性抬高，真正需要人工的只剩宪法级边界。


5. 组织无限扩展：从 O(n²) 到 O(n)
这一节从理论上说清楚，工具化层为什么能让组织规模扩展而复杂度不爆炸。
5.1 没有工具化时的复杂度增长
假设组织里有 n 个能力模块。没有 UMI 时：
•Router 需要理解每个模块的语义 → O(n) 的 Router 知识复杂度
•模块间可以相互通信 → O(n²) 的通信协议复杂度
•进化需要验证所有模块组合的行为 → O(n²) 的验证复杂度
•新模块接入需要"教" Router 和相关模块 → O(n) 的接入成本
总体：规模扩展是 O(n²) 的问题，每新增一个模块，系统总复杂度大幅上升。
5.2 工具化后的复杂度
有了 UMI + SLP + Safety Airbag：
•Router 只需理解 UMI 契约（固定复杂度）→ O(1) 的 Router 知识复杂度
•模块间通信通过标准 SLP 原语 → O(n) 的通信（每条消息有固定语义，无需协商）
•进化验证只验证 UMI 契约是否满足 → O(n) 的验证复杂度
•新模块接入只需满足 UMI 契约 → O(1) 的接入成本
总体：规模扩展是 O(n) 的问题。每新增一个满足 UMI 的模块，系统复杂度只增加常数级。

规模红利的真实基础
每增加一个经过挑战集验证的新 UMI 模块，整个组织的能力矩阵就扩展了一行——Router 无需修改，现有模块无需了解新模块，进化引擎无需重新校准。这是真正的规模红利，而不是"堆了更多 agent"。
5.3 能力矩阵：组织的能力全景图
v3.0 引入能力矩阵的概念作为组织能力的可视化工具。能力矩阵是一个自动维护的二维表：行是 SLP 原语（意图类型），列是已注册的 UMI 模块，单元格是该模块处理该意图的历史质量评分。
能力矩阵的价值不只是可视化，它直接驱动两个机制：
•Router 的精准路由：对于每个意图，Router 查询能力矩阵，选择历史评分最高的模块
•进化优先级排序：Observer 查看矩阵中评分持续下滑的单元格，确定进化优先级


6. v3.0 数据结构更新
v3.0 对 v2.0 的核心数据结构做了精确的增量修改，不推翻，只补充。
6.1 Message：intent 字段升级为 SLP 原语
v2.0 的 Message 中 intent 是 enum 类型但没有语义合约。v3.0 把它升级为 SLP 原语引用：

字段	v2.0	v3.0 变化	原因
intent	enum（自由值）	SLPrimitive（受约束原语 ID）	每个原语有固定语义合约和内嵌默认行为，接收方无需解析自然语言
payload	object（自由格式）	SLPPayload（由 intent 原语决定格式）	原语绑定了 payload schema，接收方可用机械校验而非语义理解
context_level	enum S/M/L	新增：由 module.context_requirements 自动决定，也可手动覆盖	模块声明自己需要哪个密度，减少不必要的上下文加载
airbag_check	（无）	新增：发送前自动检查操作是否需要走 Safety Airbag 流程	在消息层面提前拦截高风险操作，而非在执行层才发现
6.2 Task：新增 capability_requirements 字段
Task 现在可以声明自己需要哪些 SLP 原语能力，Planner 据此选择能满足需求的模块组合：
•capability_requirements: SLPrimitive[]  →  完成本任务需要的原语列表
•preferred_modules: string[]  →  用户或进化历史指定的首选模块（可选）
•airbag_required: AirbagLevel  →  本任务要求的最低安全工程级别
7. 角色更新：v3.0 新增与调整
v3.0 新增两个角色，并对 Router 的职责做了重要调整。

角色	变化	职责	归属层
Router	职责调整	v3.0 中 Router 不再"理解"模块，只查询能力矩阵（UMI capability_profile）做机械路由	执行层
Contract Validator ★	新增	校验新模块的 UMI 契约完整性和 airbag_spec 合规性，纯机械校验，不用 LLM	工具化层
Capability Matrix Manager ★	新增	维护能力矩阵，记录所有模块在所有 SLP 原语上的历史质量评分，供 Router 和 Observer 查询	工具化层
Challenge Curator	职责扩展	v3.0 新增职责：为每个 SLP 原语维护对应的挑战集子集，按原语类型组织测试样本	进化健康层
Meta Judge	输入来源扩展	v3.0 中 Meta Judge 使用按 SLP 原语分类的挑战集，可以分原语类型输出进化效果评估	进化健康层


8. 工具化驱动进化：v3.0 的新进化路径
工具化层不只是扩展了能力，它还为进化引擎打开了一个 v1.0 和 v2.0 都没有的维度。
8.1 原语级进化（v3.0 新增）
v1.0 和 v2.0 的进化单位是模块（模块被替换、优化或重构）。v3.0 新增原语级进化：当某个 SLP 原语在能力矩阵的整列评分都偏低时，问题不是某个特定模块，而是这个原语的语义合约定义本身存在缺陷。
原语级进化的流程：
6.Observer 发现某个 SLP 原语（如 PROPOSE_CHANGE）在能力矩阵中，所有模块的评分都低于基线
7.Evolver 生成原语合约变更提案（修改 payload 格式、调整内嵌默认行为、拆分或合并原语）
8.在沙箱中用新旧原语合约对历史消息做兼容性测试
9.通过后，新版原语合约在新消息中生效，旧消息保留旧合约（版本共存期）
10.所有模块的 intent_tags 声明更新到新原语版本，完成迁移

原语进化的意义
SLP 原语是整个通信系统的"词汇表"。如果词汇表本身不够精确，所有模块的通信质量都会受影响。原语级进化让语言本身可以进化，这是 v1.0/v2.0 不具备的能力——它们只能进化"说话的人"，不能进化"语言本身"。
8.2 接口驱动进化（Zero-Shot Module Generation 方向）
这是 v3.0 指向未来的一个方向，MVP 阶段不实现，但设计上要为它留空间。
当 Observer 发现某个 SLP 原语在能力矩阵中没有任何模块能处理（空列），当前方案是报告给用户。但 UMI 的标准化让一个更进一步的方案成为可能：Evolver 利用 UMI 契约和 SLP 原语的定义，直接生成一个新能力模块的初始版本——不是最优的，但满足 UMI 契约，可以进入沙箱测试流程。
这个方向的可行性依赖于工具化层的完整性：接口越标准化，自动生成满足接口的实现就越容易。这是工具化层为未来自组织能力打下的地基。

12. v4.0 方向：组织拓扑可进化（待 v3.0 稳定后）
ChatGPT 在 v1.0 review 中提出的"自组织型系统"，在 v3.0 完成后才有了真实的工程基础。
组织拓扑可进化需要的三个前提条件，到 v3.0 才全部满足：
•标准化接口（UMI）：拓扑变化意味着模块之间的连接关系改变，只有接口标准化之后，重新连接才不需要修改每个模块的内部实现
•可靠的进化验证（v2.0 双评估 + 挑战集）：拓扑变化比模块替换影响面更广，需要更可靠的验证机制做保底
•失败代价可控（Safety Airbag）：拓扑变化风险更高，需要 Staging 环境和 Reversibility Gate 保证拓扑变更的可逆性
v4.0 的具体方向：临时子组织（特定任务类型自动组队，完成后解散）、模块克隆（高质量配置的复制与专化）、Topology Gene（Evolver 可以提案修改角色依赖图）。但这些都是 v3.0 稳定后的事。



v3.0 设计底线
v3.0 的每一个新机制——UMI、SLP、Safety Airbag、能力矩阵——都是为了解决同一个问题：让组织的规模扩展与复杂度增长解耦。没有工具化层，组织就无法无限扩展。有了工具化层，每一个新模块都是纯粹的能力加法，而不是复杂度乘法。
