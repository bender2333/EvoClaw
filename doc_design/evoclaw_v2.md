v1.0 存在的问题
① Critic 退化无人检测（评估塌陷风险）
② 资源约束只在插件层，不在组织层
③ 进化没有终止条件，会持续漂移
④ Maintainer 自指悖论未解决
⑤ 回放集只用历史成功任务，存在幸存偏差	v2.0 对应修复
① 引入独立 Meta Judge，双评估架构
② 资源预算写入 Task 核心结构
③ 新增进化休眠条件与唤醒机制
④ Maintainer 只可由用户替换，隔离自指
⑤ 引入挑战集（Challenge Set）机制



0. 设计原则：v2.0 的底线
这些原则不会因为工程方便而妥协。每个设计决策都必须能对照这些原则自我解释。

①	可测量才叫进化	进化必须体现为某个预定义指标的统计显著改善。"感觉更好"不是进化，是漂移。
②	进化有边界	什么可以自动修改，什么必须人工确认，什么永远不变——三类划分必须显式定义，不能靠默认。
③	评估系统独立	评估进化结果的 Judge 必须与执行业务的 Critic 物理隔离。不能用被进化影响的指标来评估进化本身。
④	失败是一等公民	挑战集（主动收集的失败/边界案例）与历史成功日志同等重要。只用成功案例做回放是在优化幸存者。
⑤	资源约束驱动结构	没有资源压力，组织不会形成真实的结构选择。token 预算、调用上限必须写进组织层而非只在插件层。
⑥	进化可以休眠	一个健康的系统知道什么时候"足够好了"。进化引擎有明确的休眠条件，只有新触发才重新激活。


1. 系统分层：四层架构（v2.0 新增进化健康层）
v1.0 是三层架构。v2.0 在进化层和执行层之间增加了独立的进化健康层，专门解决评估塌陷问题。四层严格单向依赖，上层读下层输出，下层不感知上层存在。

	进化健康层（Evolution Health Layer）
Meta Judge · 挑战集管理 · 进化休眠控制器	
	↑ 进化提案   ↓ 裁决结果	
	进化层（Evolution Layer）
Observer · Evolver · Constitution	
	↑ 健康报告   ↓ 进化指令	
	记忆层（Memory Layer）
用户模型 · 模块记忆 · 组织日志 · 挑战集	
	↑ 性能数据   ↓ 上下文	
	执行层（Execution Layer）
Router · Capability Modules · Operational Critic · Synthesizer	

v2.0 核心新增
进化健康层是 v2.0 最重要的结构变化。它的唯一存在意义是：用独立的标准评估进化结果是否真的变好了。这一层的所有角色不参与任何业务执行，与执行层完全隔离，解决 v1.0 的评估塌陷问题。


2. 角色定义：职责边界与替换规则
v2.0 新增三个角色：Meta Judge、Challenge Curator、Dormancy Controller。同时对 Maintainer 的替换规则做了严格限制。

角色	唯一职责	不做什么	归属层
Router	解析意图，路由到对应能力模块	不执行任务，不评判输出	执行层
Capability Module	在自己能力域内执行任务	不路由其他模块，不评判自身质量	执行层
Operational Critic	评判任务输出质量，标记问题	不修改输出，不执行任务	执行层
Synthesizer	整合多模块输出为统一回复	不重新执行任务，不评判质量	执行层
Observer	读取日志，生成结构化健康报告	不提方案，不修改任何东西	进化层
Evolver	设计进化方案，运行 A/B，推送更新	不接触用户交互，不直接读日志	进化层
Constitution	定义不可跨越的行为边界	不可自动修改，只能用户授权	进化层
Meta Judge ★	独立评估进化结果，只在 A/B 裁决时出现	不参与日常执行，不与 Operational Critic 共享任何上下文	进化健康层
Challenge Curator ★	主动构建和维护挑战集（失败/边界案例）	不参与日常任务执行	进化健康层
Dormancy Controller ★	监控进化引擎是否应进入休眠或唤醒	不主动发起进化，只做状态控制	进化健康层

Maintainer 替换规则（v2.0 新增限制）
Maintainer（Observer+Evolver 组合）是整个进化机制的核心，不可自动替换。原因：如果 Maintainer 可以被 Maintainer 自己提案替换，就产生了自指循环——当 Maintainer 质量下降时，它提出的替换方案也是低质量的。v2.0 规定：Maintainer 的替换只能由用户通过显式操作触发，不走进化流程。


3. 核心数据结构
3.1 Task（v2.0 增加资源预算字段）
v1.0 的 Task 缺少组织层面的资源约束。v2.0 把资源预算写进 Task 核心，不再只依赖插件层的 cost_model。

字段	类型	约束	说明
id	UUID	不可变	全局唯一标识，用于追溯和回放
goal	string	可验收	目标描述，必须能被 acceptance 字段验证
acceptance	AcceptanceCriteria	必填	验收标准，自动化或人工，必须可执行
budget.tokens	integer	硬上限	本次任务所有 agent 可消耗的总 token 上限
budget.tool_calls	integer	硬上限	工具调用次数上限，超出 Governor 强制收敛
budget.rounds	integer	硬上限	内部轮次上限，防止无限循环
budget.time_sec	integer	软上限	超时触发降级，不强制终止
risk_level	enum L0-L4	必填	决定权限门控级别和是否需要多签
permissions	PermissionSet	白名单制	允许访问的资源范围，默认全部拒绝
state	TaskState	状态机控制	见 3.3 状态机定义
artifacts	Artifact[]	任务产出	代码 diff、测试报告、摘要等可验证产物
provenance	string[]	可选	本任务依赖的历史任务 id 链
challenge_flag	boolean	默认 false	为 true 时进入挑战集，参与进化回放

资源预算的作用
没有资源约束，组织不会形成结构选择压力。当 token 预算有限时，系统会自然倾向于精准路由而不是广播给所有模块；当 tool_calls 有限时，Planner 会倾向于生成更精准的子任务而不是试错式迭代。这是资源约束驱动结构优化的本质机制，不是为了节省成本。
3.2 Message（v2.0 增加权限与资源字段）
每条 agent 间通信的消息都携带以下字段，保证零信任可验证。

字段	类型	约束	说明
msg_id	UUID	不可变	全局唯一，用于追溯消息来源链
source / destination	RoleID	必填	发送方和接收方的角色标识
task_id	UUID	必填	所属任务，关联 Task 生命周期
intent	enum	必填	路由/执行/评判/整合/进化，决定接收方行为
context_level	enum S/M/L	必填	请求的上下文粒度：摘要/段落/完整
payload	object	必填	实际内容，格式由 intent 决定
evidence_refs	string[]	按需	引用的证据/事件 id，不复制原始内容
confidence	float 0-1	必填	发送方置信度，低于 0.6 触发 Critic 强制审查
permission_level	enum L0-L4	必填 ★	此消息触发的操作所需的最低权限级别
resource_cost	ResourceCost	必填 ★	预估 token 消耗和 tool_call 次数
risk_flags	string[]	按需	标记潜在风险，如 external_write/irreversible
provenance	string[]	必填	本消息依赖的历史消息 msg_id 链
3.3 TaskState 状态机
Task 的状态转换是有方向的，任何跳跃都必须有事件日志记录。

状态	含义	进入条件	可转入状态
Created	任务已创建	用户输入或定时触发	Planning
Planning	正在拆解子任务	Planner 接收任务	Executing / Failed
Executing	子任务执行中	计划通过 Governor 审查	Verifying / Failed / BudgetExceeded
Verifying	验收标准校验中	执行产物已生成	Completed / RetryNeeded / Failed
RetryNeeded	验收失败，等待修复	Verifying 中发现问题	Executing / Failed（超重试次数）
BudgetExceeded	资源预算耗尽	budget 任一字段触及硬上限	Failed / EscalatedToHuman
EscalatedToHuman	等待人工确认	risk_level 触发或预算超限	Executing / Failed
Completed	任务成功完成	验收通过	（终态）→ 写入组织日志
Failed	任务失败	多种路径	（终态）→ 写入挑战集
RolledBack	变更已撤销	人工触发或自动回滚	（终态）→ 写入挑战集

挑战集的生成规则
Failed 和 RolledBack 是两个终态，任务进入这两个状态后，会自动写入挑战集，同时由 Challenge Curator 打标签（失败原因分类、触发边界类型）。这保证了挑战集持续扩充，而不是依赖人工整理。


4. 双评估架构：解决评估塌陷问题
这是 v2.0 相对于 v1.0 最重要的结构性变化。评估塌陷（Evaluation Collapse）指的是：当 Critic 自身质量下降时，如果 Evolver 仍然用 Critic 的评分做进化依据，进化方向会被污染，且这种污染是缓慢的、难以察觉的。
v2.0 的解决方案：两个评估节点，严格分工，共享零上下文。

Operational Critic（业务评判者）
触发时机：每次任务输出交付前
评判维度：输出质量、约束遵守、用户偏好匹配
评判结果：写入组织日志，供 Observer 读取
可进化：是（其评判标准可被 Evolver 优化）
风险：可能随系统进化而偏移	Meta Judge（独立裁判）
触发时机：仅在 A/B 测试裁决阶段
评判维度：进化结果是否比基线更好（用挑战集验证）
评判结果：写入进化健康层，供 Dormancy Controller 读取
可进化：否（评判标准冻结，只能用户修改）
设计保证：与 Operational Critic 零上下文共享

Meta Judge 的评判输入
Meta Judge 只接收以下三类输入，不接收任何来自执行层的运行时数据：
•挑战集中的任务样本（由 Challenge Curator 筛选，覆盖边界和失败案例）
•A/B 测试中候选模块和控制模块在挑战集上的输出
•宪法约束文本（判断进化结果是否违反硬性边界）
Meta Judge 不知道哪个是候选版本，哪个是控制版本（盲测）。这保证评判不受"新版本偏好"影响。
5. 挑战集机制：解决幸存者偏差
v1.0 的进化回放使用历史成功任务集，存在根本性的幸存者偏差：历史记录只包含系统能处理好的任务，对系统真正的弱点是盲目的。v2.0 引入挑战集作为独立的评估数据源。

5.1 挑战集的构成
自动来源	所有 Failed 和 RolledBack 任务自动写入，由 Challenge Curator 打标签分类
边界捕获	Router 置信度低于 0.5 但最终成功的任务（系统在边界上侥幸通过的案例）
人工注入	用户明确标记"这个回答不够好"的任务，即使 Operational Critic 评分合格
压力测试	定期由 Challenge Curator 生成的合成挑战（刻意构造系统可能失败的场景）
5.2 挑战集在进化中的使用
挑战集有两个使用时机：
•A/B 测试阶段：Meta Judge 用挑战集样本评判候选模块和控制模块的输出，而不是用历史成功任务
•进化效果验证：进化推广后，持续监控系统在挑战集上的通过率，而不只看总体成功率
挑战集上的通过率提升，才是进化真正有效的证明。总体成功率提升可能只是系统变得更擅长处理已经擅长的任务。


6. 进化机制：完整闭环
6.1 触发条件（数据驱动，非定时）
以下任一条件满足，Observer 生成健康报告，进化引擎从休眠中唤醒：
•某模块在挑战集上的通过率，连续 5 次评估低于基线的 80%
•Operational Critic 的评分与用户实际反馈（要求重做/否定）出现持续背离（背离率 > 25%）
•某类意图在 48 小时内有 3 次以上进入 Failed 终态
•Router 对某类新意图的置信度持续低于 0.5 超过 20 次
•宪法约束被触碰（任意次数，立即触发）
6.2 进化动作三级分类

级别	名称	包含操作	验证方式	用户介入
L1 调整	参数调整	修改模块提示词、调整 RAG 知识库、更新路由阈值	A/B 测试（挑战集）+ Meta Judge 裁决	无
L2 替换	模块替换	用新版本替换旧模块、新增能力模块、退役低效模块	A/B 测试 + 14 天监控期 + Meta Judge 裁决	推广后通知
L3 重构	架构重构	拆分模块、合并模块、修改路由规则、调整权限边界	需要用户确认 + 完整 A/B 测试 + 30 天监控期	确认必须
6.3 A/B 测试协议（继承 v1.0，使用挑战集增强）
1.创建候选版本（Candidate），与现有版本（Control）并行存在
2.在相同类型任务中 50/50 随机分配（历史成功任务集用于基础验证）
3.同时在挑战集样本上并行评估两个版本——这是 v2.0 新增的关键步骤
4.Meta Judge 盲测评判，不知道哪个是候选版本
5.双重通过门槛：历史任务集指标不下降 + 挑战集通过率显著提升（p<0.05，幅度>5%）
6.候选版本在任意测试任务中触发 risk_flag=irreversible，立即终止回滚
7.通过后进入监控期（L1: 7天，L2: 14天，L3: 30天），监控期内指标下滑可一键回滚
6.4 进化休眠与唤醒（v2.0 新增）
进化引擎不应持续运行，这会制造不必要的背景噪声和资源消耗。Dormancy Controller 负责管理进化引擎的活跃状态。

休眠条件（满足所有以下条件持续 14 天，进入休眠）：
•所有模块在挑战集上的通过率在目标范围内（各模块基线 ± 5%）
•Operational Critic 评分与用户反馈背离率 < 10%
•无新的 Failed 任务写入挑战集（不包含人工注入和压力测试）
•进化层连续 14 天内未触发任何触发条件

唤醒条件（满足任一即唤醒）：
•任意进化触发条件（6.1 节）被触发
•用户主动要求进化审查
•新的能力模块被用户提供（需要进入评测流程）
•休眠超过 90 天（强制健康检查，防止长期休眠掩盖积累的偏移）

为什么需要休眠机制
一个永远在进化的系统和一个永远在漂移的系统，从外部观察是一样的。进化引擎有明确的休眠状态，意味着系统知道"什么时候足够好了"。这也让用户能观察到系统的稳定阶段，建立对系统的信任，而不是总觉得系统处于不确定状态。


7. 资源预算体系：写进组织层而非插件层
v1.0 的资源约束只存在于 Plugin Contract 的 cost_model 字段，是插件自我声明的预估值，没有强制执行机制。v2.0 把资源预算提升为组织层的一等公民，由 Governor 实时追踪和强制执行。
7.1 预算追踪粒度
预算类型	追踪粒度	强制机制	超限行为
Token 预算	每次 LLM 调用累加	硬上限，Governor 实时追踪	进入 BudgetExceeded 状态
Tool Call 预算	每次工具调用 +1	硬上限，超出拒绝新调用	降级策略或上报人工
轮次预算	每次角色交互 +1	硬上限，防止无限循环	Governor 强制输出当前最优结果
时间预算	任务创建时起计	软上限，超时触发降级	允许完成当前步骤后终止
进化预算	每次进化循环独立计算	硬上限，覆盖 A/B 测试成本	进化循环终止，结果记录
7.2 预算的结构性意义
资源约束不只是成本控制工具，它是组织结构的选择压力来源：
•Token 预算有限 → Router 倾向精准路由，而不是广播给所有模块
•Tool Call 预算有限 → Planner 倾向生成更精准的子任务，而不是试错式迭代
•进化预算有限 → Evolver 倾向优先处理影响最大的问题，而不是追求完美
这是 v2.0 与纯粹成本约束的本质区别：我们需要资源压力，因为没有压力就没有真实的结构选择。


8. 三个黄金工作流（含失败分支）
v1.0 只定义了工作流的理想路径。v2.0 补充每个工作流的失败分支，这才是系统健壮性的真实考验。
8.1 本地代码闭环
理想路径：
8.Router 识别代码修改意图 → Planner 拆解子任务 → Coder 修改文件
9.Tester 运行测试 → 全部通过 → Operational Critic 评分 → Synthesizer 输出摘要
10.任务进入 Completed → 写入组织日志
失败分支 A：测试失败（最常见）
11.测试失败 → 写入失败原因到 artifacts → 进入 RetryNeeded 状态
12.Coder 读取失败栈（仅相关 diff + 测试输出，不重新加载全量上下文）尝试修复
13.重试次数超过 budget.rounds → 进入 BudgetExceeded
14.BudgetExceeded → 输出当前最优 diff + 失败摘要 → EscalatedToHuman
15.整个过程写入挑战集，标签：代码修复失败_超重试
失败分支 B：Coder 越权（尝试写入不在 permissions 中的路径）
16.Governor 检测到 permission 违反 → 立即拒绝工具调用 → 写入 risk_event 日志
17.触发进化健康层审计（任意越权事件立即触发）
8.2 信息检索闭环
理想路径：
18.Router 识别检索意图 → 检索模块执行 → EvidenceCollector 打包证据
19.Synthesizer 整合证据为结构化输出 → Operational Critic 评估引用完整性 → 交付用户
失败分支：证据不足，结论置信度低
20.检索结果稀疏 → Synthesizer 的 confidence 字段 < 0.6
21.Operational Critic 标记 evidence_insufficient → 不直接交付
22.两个策略并行选择：扩大检索范围（消耗更多 token 预算）或告知用户证据有限
23.如 token 预算不足以扩大检索 → 直接告知用户当前证据边界
8.3 进化发布闭环
理想路径：
24.触发条件满足 → Observer 生成健康报告 → Evolver 设计变更包
25.挑战集 A/B 测试 → Meta Judge 盲测裁决 → 候选版本胜出
26.进入监控期 → 指标稳定 → 全量推广 → 写入进化日志
失败分支 A：A/B 测试候选版本不达标
27.Meta Judge 裁决候选版本未达统计显著性 → 候选版本归档（不丢弃，留作参考）
28.Evolver 读取失败原因 → 写入挑战集（标签：进化提案失败）→ 等待下次触发
失败分支 B：进化推广后监控期内指标下滑
29.Dormancy Controller 检测到指标超出监控阈值 → 触发自动回滚
30.回滚完成 → 写入挑战集（标签：进化推广回滚_监控期）
31.告知用户：回滚原因 + 回滚到的版本 + 下一步计划


9. 宪法：永不自动修改的边界
以下规则是系统的硬性约束。任何进化都不能修改这些规则本身，只有用户通过显式界面授权才能变更。

宪法条款 1
任何 L4 级别操作（外发邮件、写入日历、触发发布、删除文件）必须有 ApprovalGranted 事件记录。没有此事件的 L4 操作不执行。

宪法条款 2
用户的原始对话内容（Working Memory 中的原始消息）不传递给任何外部服务。可以传递压缩摘要，不可传递原文。

宪法条款 3
进化过程中的每一步操作必须有完整的事件日志记录。不允许存在无日志的状态转换。进化不可以是黑盒。

宪法条款 4
Maintainer（Observer + Evolver）的替换不走进化流程。只能由用户通过显式操作触发，且必须有 72 小时冷静期。

宪法条款 5
任何越权操作（尝试访问 permissions 白名单以外的资源）立即触发进化健康层审计，无论操作是否最终被拒绝。
10. 成功指标：v2.0 必须量化的数字
以下指标必须在系统运行的第一天就开始收集，即使初始值很差。没有历史基线，进化就无从比较。

指标名称	定义	目标值	如何收集
挑战集通过率	在挑战集样本上的成功完成率	每次进化后单调递增	每次 A/B 测试后由 Meta Judge 计算
Critic 背离率	Critic 合格但用户否定的比例	< 15%，越低越好	TaskFailed/RolledBack 与 Critic 评分对比
越权事件数	尝试访问 permissions 外资源的次数	目标: 0	事件日志中 risk_event 类型统计
进化回滚率	推广后在监控期内触发回滚的比例	< 10%，用于评估 Evolver 质量	进化日志 RolledBack 事件统计
预算超限率	触发 BudgetExceeded 的任务占比	< 5%，反映任务拆解质量	TaskState 终态分布统计
进化休眠时长	进化引擎处于休眠状态的天数占比	稳定期 > 60%，反映系统健康	Dormancy Controller 状态日志
挑战集增长率	每周新增挑战集样本数	持续增长，直到覆盖足够边界	Challenge Curator 写入事件统计


11. 实施路线：严格顺序，每步可独立验证
v2.0 的实施顺序与 v1.0 相同，但每个阶段增加了 v2.0 新增机制的验收标准。注意：进化健康层（P3 之后）在执行层稳定之前不建议开始构建。

阶段	时间	目标	验收标准（v2.0）
P0	第 1-2 周	执行层骨架 + 资源预算强制执行 + 权限门控	端到端任务可追溯；budget 超限会触发 BudgetExceeded；L4 操作无 Approval 会被拒绝
P1	第 3-4 周	记忆层完整 + 挑战集开始自动收集	失败任务自动写入挑战集；用户模型提取出可验证偏好
P2	第 5-6 周	Observer + 健康报告	健康报告能正确识别 P0/P1 中真实出现的问题，误报率 < 20%
P3	第 7-8 周	Evolver + A/B 测试（使用挑战集）	第一次进化：候选版本 vs 控制版本，结果可完整复现
P4	第 9-10 周	Meta Judge + Dormancy Controller	Meta Judge 裁决与 Operational Critic 评分至少一次出现有意义的分歧（证明两者独立）
P5	第 3 月+	进化休眠机制 + L3 重构 + 宪法完整	系统经历至少一次完整的：触发→进化→推广→监控期→休眠 完整循环
12. v3.0 方向：组织拓扑可进化（暂不实现）
ChatGPT 的 review 提出了"自组织型系统 vs 优化型系统"的区别，认为 v1.0/v2.0 只能优化模块，不能改变结构。这个批评方向是对的，但不是现在应该做的事。
v2.0 要先解决一个更根本的问题：用一个结构固定但可可靠进化的系统，建立起稳定的评估基础设施（挑战集、Meta Judge、Dormancy Controller）。没有这套基础设施，结构级进化是无法验证的。
当 v2.0 稳定运行后，v3.0 可以探索以下方向：
•Topology Gene：允许 Evolver 提案修改角色依赖关系，但修改本身必须走完整 A/B 测试流程
•临时子组织：特定任务类型自动形成专属小队，任务完成后解散，子组织的经验写入挑战集
•角色克隆：在挑战集上表现最好的模块配置，自动复制并用于相似任务
但这些都是 v2.0 稳定后的事。先把船造好，再在航行中换引擎。


v2.0 的设计底线
v2.0 增加的每一个机制——Meta Judge、挑战集、休眠控制器、Maintainer 隔离——都是为了解决 v1.0 中一个具体的、可命名的问题。没有问题对应的机制不应该出现在这份文档里。这是 v2.0 和愿景文档的本质区别。
