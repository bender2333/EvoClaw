
EvoClaw
以组织进化为核心的 AI 管家系统设计文档

0. 设计哲学：先定义清楚几个词
在正式设计之前，需要对三个核心概念做精确定义，而不是模糊地使用它们。这些定义将贯穿整份文档。
0.1 "进化"是什么
进化在本文档中只有一个含义：某个可测量指标在时间轴上显著改善，且这种改善来自系统内部的自动调整，而不是人工手动修改配置。
不符合这个定义的情况：用户手动添加了一个新插件、开发者更新了代码、系统换了底层模型。这些是升级，不是进化。
0.2 "组织"是什么
组织在本文档中指：多个 agent 按照明确的角色分工和通信协议协作，使得整体能力 系统性地 超过任意单个 agent 的能力上限。
关键词是"系统性地"——不是偶尔，不是靠运气，而是在结构层面保证的能力放大。
0.3 "插件/能力模块"是什么
每个能力模块是一个完整的合约包，包含四个要素：系统提示词（定义角色和行为边界）、工具集（这个模块被允许调用的外部工具）、性能基线（判断这个模块好不好的量化标准）、版本记录（每次进化的 diff 和原因）。
插件不是一段代码，不是一个 API，而是以上四个要素的组合体。替换插件意味着替换这整个合约包。

核心设计原则：每一个设计决策都必须能回答"如果这里失败了，会发生什么？"如果回答是"系统崩溃"或"不知道"，这个设计就需要重做。


1. 系统架构：三层分离
EvoClaw 的整体架构分为三层。这三层有严格的单向依赖关系：上层感知下层的输出，但下层不知道上层的存在。这保证了每一层可以独立迭代，不会产生循环依赖。

	第三层：进化层 (Evolution Layer)
Observer · Evolver · Constitution
读取健康数据，提出并验证进化方案	
	↑ 健康报告        ↓ 进化指令	
	第二层：记忆层 (Memory Layer)
用户模型 · 模块记忆 · 组织日志
多粒度记忆存储，为上下两层提供上下文	
	↑ 性能数据        ↓ 用户偏好/上下文

Router · Capability Modules · Critic · Synthesizer

	↑ 消息输入        ↓ 回复输出

2. 角色定义：职责边界必须清晰
每个角色只做一件事，并且明确定义它不做什么。这是防止角色漂移的唯一方式。

2.1 执行层角色
角色	唯一职责	不做什么
Router（路由器）	解析用户意图，决定调用哪个能力模块	不执行任务，不修改输出，不做判断
Capability Module（能力模块）	在自己的能力域内执行任务并返回结果	不路由其他模块，不评判自己的输出质量
Critic（评判者）	在输出交付用户前评估质量，标记问题	不修改输出内容本身，不执行任务
Synthesizer（整合者）	把多个模块的输出整合成统一回复	不重新执行任务，不做质量评判

关键设计：Critic 的激励与 Capability Module 必须对立。Module 优化任务完成率，Critic 优化错误发现率。这种内建张力保证系统不会退化成"所有人都说好"的回音壁。

2.2 记忆层（数据结构，非角色）
记忆层不是 agent，而是一套分层数据结构，为上下两层提供服务。
•工作记忆（Working Memory）：当前会话的完整上下文。会话结束后压缩提炼，丢弃原始记录。
•用户模型（User Model）：从交互中结构化提炼的偏好。例如："偏好简洁回复"、"财务类任务需要确认"、"早上8点前不发提醒"。永不存储原始对话。
•模块记忆（Module Memory）：每个能力模块的 RAG 知识库。包含领域知识和历史执行模式。
•组织日志（Org Log）：每次任务的执行记录：模块版本、耗时、Critic 评分、用户反馈信号。这是进化层的原始数据来源。

2.3 进化层角色
角色	唯一职责	不做什么
Observer（观察者）	定期读取组织日志，生成健康报告：哪个模块指标下滑，哪类任务失败率上升	不提出解决方案，不直接修改任何东西
Evolver（进化引擎）	读取健康报告，设计进化方案，运行 A/B 测试，验证后推送更新	不接触用户交互，不观察日志（通过 Observer 的报告间接感知）
Constitution（宪法）	定义所有角色都不能跨越的行为边界，以及什么级别的进化需要用户确认	宪法本身不可自动修改，必须经用户授权


3. 进化基因：什么东西被修改
这是整个系统最关键的设计决策。"自我进化"必须回答一个具体问题：修改的是什么？不同的修改对象有完全不同的风险和工程复杂度。

进化基因	内容	可修改方式	验证方法
系统提示词	定义模块角色、行为约束、输出格式	增量 diff 修改，保留版本历史	A/B 测试：新旧版本并行跑，评判 agent 打分
工具集	模块被允许调用的外部工具清单	增加/移除工具，调整调用参数	沙箱测试新工具，验证副作用边界
路由规则	Router 的意图分类逻辑	调整分类阈值，新增意图类别	回归测试：历史意图集的分类准确率不下降
模块记忆	RAG 知识库中的领域知识	追加新知识，标记旧知识过期	检索相关性评分，与 Critic 结合验证

明确排除在自动进化范围之外的东西：底层模型的权重（不做微调）、系统整体架构（需要用户确认）、宪法规则（需要用户授权）。这三类修改的成本和风险远超收益。
4. 通信协议：信息的骨架
这是 EvoClaw 相对于 OpenClaw 最核心的工程创新。所有 agent 之间的消息必须遵循统一的结构化格式，而不是自由文本传递。
4.1 消息结构
每条消息携带以下字段：
•msg_id：全局唯一 ID，用于追溯消息来源链
•source / destination：发送者和接收者的角色标识
•task_type：任务类别（路由、执行、评判、整合、进化），决定接收方如何处理
•context_request：接收方声明需要哪个粒度的上下文（摘要级 / 段落级 / 完整级）
•payload：实际内容，格式由 task_type 决定
•provenance：这条消息依赖了哪些历史消息的 msg_id
•confidence：发送方对自身输出的置信度评分（0-1），低置信度消息会触发 Critic 强制审查
4.2 多粒度上下文（解决 context 爆炸）
每个任务完成后，系统自动生成三个粒度的摘要，存入记忆层：
•一行摘要（≤20字）：用于快速路由判断
•段落摘要（≤150字）：用于需要背景的模块
•完整记录：只在明确需要细节时展开，按需加载
下游 agent 在消息的 context_request 字段声明需要哪个粒度，系统按需提供。这是 lazy loading 在 AI 通信中的直接类比。
4.3 来自 OpenClaw 的继承
执行层的用户交互部分直接继承 OpenClaw 的设计精华：实时响应、多平台支持、自然语言理解、任务透明展示。通信协议只作用于 agent 之间的内部通信，对用户界面透明。


5. 进化机制：从数据到改变的完整闭环
这一节定义了进化如何真实发生，而不是概念层面的描述。每一步都有明确的触发条件、执行动作和验收标准。
5.1 触发条件（什么时候进化）
进化不是定时器触发的，而是由数据触发的。以下任何一个条件满足时，Observer 生成健康报告，Evolver 启动进化流程：
•某个能力模块的 Critic 评分，在连续 10 次任务中低于该模块历史均值的 80%
•某类意图的路由准确率，在滑动窗口内低于 90%
•用户反馈中，负向信号（明确否定、要求重做）在 48 小时内超过 3 次
•新的任务类型出现，且没有现有模块能覆盖（Router 的置信度持续低于 0.5）
5.2 进化动作（能做什么）
Evolver 可以执行三类进化动作，复杂度和风险递增：
•调整（Adjustment）：修改现有模块的系统提示词，或调整 RAG 知识库。自动执行，用户不感知。
•替换（Replacement）：用新版本模块替换旧版本，需经 A/B 测试验证。自动执行，完成后在系统日志中记录。
•重构（Restructuring）：新增模块、拆分模块、合并模块、调整路由规则。需要用户简单确认后执行。

用户只需参与"重构"级别的进化决策。这才是真正的"人类稀疏参与"——不是让用户每次二选一，而是只在架构变更时出现。
5.3 A/B 测试协议（如何验证进化好不好）
任何模块级别以上的进化方案，都必须经过 A/B 测试。测试协议如下：
1.用新方案创建候选模块（Candidate），与现有模块（Control）并行存在
2.Router 在接下来的相同类型任务中，随机以 50/50 比例分配给两个版本
3.独立的 Critic 对两个版本的输出分别打分，不知道哪个是候选版本（盲测）
4.累积 15 个有效评判后，计算统计显著性。候选版本评分显著高于控制版本（p<0.05），且优势幅度超过 5%，则推广
5.如果候选版本在任何测试任务中导致严重错误（Critic 评分低于 0.3），立即终止测试并回滚
5.4 失败驱动而非成功驱动
EvoClaw 进化的核心燃料是失败分析，而不是成功积累。每次任务完成后，系统都运行反事实推理："如果这次失败了，最可能的原因是什么？"
即使任务成功，这个推理也会暴露潜在脆弱点。一个系统通过 100 次任务而从未失败，但每次都在边缘情况徘徊，和一个真正稳健的系统是完全不同的。只有失败分析能区分这两种情况。


6. 组织能力放大：为什么多 agent 比单 agent 更强
这一节回答一个根本问题：组织为什么能超越个体？不是靠人数，而是靠结构性的能力放大机制。
6.1 张力驱动质量
如前所述，Critic 和 Capability Module 拥有不同的评价函数。这种内建冲突保证了即使面对相同的任务，系统会从两个对立的角度审查，而不是线性地执行。
这对应人类组织中"审计部门独立于业务部门"的设计。没有独立审计，业务部门会自然优化"看起来完成了任务"而不是"真正完成了任务"。
6.2 专业化带来深度
一个通用 LLM 处理邮件、代码、财务、日程，每个领域都是泛泛而谈。一个专门的邮件模块，其 RAG 知识库里存储了用户过去三个月的沟通风格、常用表达、重要联系人偏好，输出质量是单个通用 LLM 无法达到的。
这对应人类组织中分工的本质价值：不是让更多人做同一件事，而是让每个人在自己的领域做得更深。
6.3 信息架构超越上下文限制
单个 LLM 的上下文窗口是硬限制。但通过多粒度记忆架构，EvoClaw 的"有效记忆"远超任何单个 agent 的上下文窗口：Router 只读一行摘要就能路由，专业模块只读自己领域的知识，Evolver 读的是压缩后的性能指标。
整个组织共同持有的信息量，等于所有模块记忆的叠加，而没有任何单个节点需要承载全量信息。这正是人类组织超越个体记忆的方式。
6.4 规模红利：每个验证过的新模块让整体变强
当一个新的能力模块通过 A/B 测试被加入系统，受益的不只是这个模块负责的任务类型。Router 的意图分类能力随之扩展，Synthesizer 整合多模块输出的能力随之增强，Evolver 有了更多的进化素材。
这是真正的组织规模红利，而不是简单的 1+1 叠加。


7. 用户视角：你感知到的是什么
用户不需要理解系统内部如何运作。但用户应该能清楚地感知系统在进步。
7.1 日常交互体验
你和 EvoClaw 的日常交互，与使用 OpenClaw 没有区别：在 Telegram 或飞书里发消息，它立刻回复，帮你处理邮件、日程、搜索、文档等。
唯一的不同是：它会逐渐变得更懂你。不是因为你告诉了它什么，而是因为它在观察和提炼。
7.2 你唯一需要做的事
系统设计了两种反馈机制，极低负担：
•自然反馈：你正常使用时的行为就是反馈。要求重做某个任务、修改某个输出、忽略某个提醒——这些行为都被 Observer 读取。你不需要点赞踩或填问卷。
•架构确认：当系统准备做重构级进化时（例如增加一个新的专业模块），会用一句话告诉你它打算做什么，等你点确认。这种交互每个月不会超过两三次。
7.3 可观察的进化时间线
起步阶段	和 OpenClaw 一样好用，24/7 响应，覆盖日常事务。此时系统在收集基线数据。
2-4 周	用户模型积累了足够数据，开始显现个性化：邮件回复更符合你的风格，提醒时机更准确，错误率明显下降。
1-3 月	第一次重构：系统提议增加新的专业模块（基于观察到的高频未覆盖需求）。你一键确认后，新能力立即可用。
3 月以上	组织已经经历多轮进化。能力覆盖面与深度都远超初始状态。你会感觉到"这不是我最初安装的那个系统"。


8. 实施路线：按顺序，每步可验证
EvoClaw 的构建必须分阶段，每个阶段结束时有清晰的验收标准。不能在没有验证执行层的情况下开始构建进化层。

阶段	时间	目标	验收标准
P0	第 1-2 周	搭建执行层骨架：Router + 2个核心模块（邮件/日程）+ Critic + 通信协议	能完成端到端任务，消息链路可追溯，Critic 能发现至少 30% 的明显错误
P1	第 3-4 周	接入记忆层：用户模型开始积累，工作记忆压缩可用，组织日志开始记录	用户模型在 20 次交互后能提取出至少 3 个有效偏好
P2	第 5-6 周	实现 Observer：定期读取日志，生成结构化健康报告	健康报告能识别出真实存在的性能问题，误报率 < 20%
P3	第 7-8 周	实现 Evolver 的调整级进化：自动修改模块提示词并 A/B 测试	第一次自动进化完成，且可以通过日志完整复现进化过程
P4	第 2-3 月	扩展能力模块至 5 个以上，实现替换级进化，Synthesizer 可用	多模块协同任务的质量高于任意单模块的独立输出
P5	第 3 月+	开放重构级进化，宪法框架完整，用户确认流程可用	系统在无人工干预下，完成至少 1 次经用户确认的架构级进化
9. 已知风险与硬性约束
9.1 需要监控的风险
•进化漂移：Evolver 经过多轮优化后，系统行为偏离初始设计意图。缓解：宪法定期审查，Observer 的健康报告包含意图对齐度评分。
•Critic 退化：如果 Critic 的评判标准也在进化，可能导致评判质量下滑，进而进化方向错误。缓解：Critic 的评判标准单独版本控制，变更需要更高的验证门槛。
•A/B 测试污染：候选模块和控制模块处理了不同类型的任务（不是随机分配），导致测试结论无效。缓解：严格的任务类型匹配，同类任务才进入同一测试组。
•计算成本膨胀：模块数量增加导致每次任务调用的总成本线性增长。缓解：Router 优先精准路由，避免不必要的多模块并发调用。
9.2 宪法级硬性约束（不可进化的边界）
•任何模块不可以在没有用户明确授权的情况下执行财务交易
•用户的原始对话记录不可以被传递给任何外部服务
•进化过程必须完整可审计，不可以有"黑盒"进化步骤
•宪法本身的修改必须经用户通过显式界面确认，且有 24 小时冷静期

这份设计文档的价值不在于它描述了一个多么宏大的系统，而在于它的每一个设计决策都可以被质疑、被测试、被推翻。没有魔法，只有可以验证的机制。
