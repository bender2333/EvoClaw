# Product Requirements Document - EvoClaw

**Author:** bender
**Date:** 2026-02-25
**Last Edited:** 2026-02-26
**Edit Summary:** 架构回溯同步（PRD-SYNC-1/2/3）+ 原始设计文档补充（FR69-FR76：ε-greedy路由/Pattern抽象/9步Toolization/进化预算/Plugin必填字段/三集评估/SLP梯度压缩/Adapter演进/Maintainer隔离）+ 现有FR丰富（FR9/22/23/24/33细节补充）

## 1. 执行摘要

### 1.1 产品概述

EvoClaw 是一个以组织进化为核心的 AI 管家系统。对用户而言，它是一个统一人格的智能管家（"贾维斯"）；在底层，它是一个由多个 AI Agent 组成的自进化组织，能够自动发现协作模式、将复杂协作编译为轻量工具、淘汰低效模块，持续放大整体智能。

### 1.2 核心问题

单个 AI Agent 的能力存在天花板——context window 是 AI 的"邓巴数"。现有多 Agent 方案（如 OpenClaw）实现了 Agent 协作，但存在三个根本缺陷：组织结构静态，无法自主进化；记忆碎片化，跨会话经验无法积累和复用；通信低效，Agent 间全量传递 context 导致成本膨胀和信息衰减。

### 1.3 设计哲学

- **忒休斯之船（彻底解耦）**：系统中没有不可替代的"超级主脑"。从网关、通信总线到具体 Agent 和工具集，全部是独立节点，任何组件可在不停机的情况下被替换、升级或销毁
- **高保真通信（渐进式形式化）**：Agent 间通信采用混合协议——可量化的约束由形式化验证引擎直接验证，不可量化的语义保留自然语言按需评估。通信由"小脑"自动处理打包、验证、冲突检测，LLM 只看精炼摘要，最大化释放 context 窗口。约束模板持续积累，通信效率随使用自动提升
- **降维固化（真正的进化）**：组织进化的最终目的不是维持庞大团队，而是将复杂的多 Agent 协作过程"编译"为单一的、低成本的轻量级工具

### 1.4 解决方案

EvoClaw 采用基于高保真事件总线的扁平化动态拓扑，由四大核心组件构成：

- **极速前台（Fast Router / System 1）**：用户唯一触点，在已固化的工具库中匹配已知问题秒回；未知问题抛给后台组织
- **组织通信总线**：通信中枢，所有 Agent 通过发布/订阅协作。采用混合协议，形式化约束由验证引擎直接验证，语义约束由 LLM 按需评估，小脑自动处理打包与冲突检测，保证组织扩大时信息不衰减
- **动态专家集群（Dynamic Holons / System 2）**：无状态原子 Agent 临时组队（规划者、分析师、审查员），处理复杂未知任务的多步推理和博弈
- **进化编译引擎（Compiler）**：独立于业务的旁路监听者。观察总线上的任务流转，发现重复协作模式后自动编译为轻量插件，注册到前台，释放集群算力

### 1.5 进化生命周期（Toolization Loop）

能力进化的四阶段闭环：
1. **涌现**：复杂新需求由专家集群昂贵协作完成
2. **模式识别**：编译引擎发现同类任务被多次以一致路径成功执行
3. **降维编译**：剥离多余思考过程，将多 Agent SOP 编译为独立插件
4. **替换释放**：新插件经沙盒测试后热更新，下次同类需求由前台秒回，集群算力释放给下一个未知难题

### 1.6 分层记忆体系

仿脑结构的多层记忆，由"小脑"自动管理，LLM 只看精炼摘要：
- **不可变事件账本**：一切派生数据的唯一真实来源
- **工作记忆**：当前会话上下文，会话结束后压缩提炼
- **情景记忆**：历史任务记录，按日期存储
- **语义记忆**：提炼的知识——Agent 画像、协作模式，永久存储
- **用户模型**：结构化用户偏好画像
- **挑战集**：失败/边界案例，进化的核心数据源

### 1.7 核心差异化

| 维度 | 现有方案（OpenClaw 等） | EvoClaw |
|------|------------------------|---------|
| 架构拓扑 | 金字塔层级 | 扁平化事件总线 + 四大组件 |
| 组织结构 | 静态，开发者预设 | 动态，自我进化，忒休斯之船式热替换 |
| 响应模式 | 统一处理 | System 1 秒回 + System 2 深度推理双通道 |
| 新能力获取 | 人工开发插件 | 自动从协作模式编译（Toolization Loop） |
| 通信方式 | 文本拼接，全量 context 传递 | 混合协议，形式化+语义约束共存，渐进式演化 |
| 记忆系统 | 单会话或简单持久化 | 仿脑分层记忆 + 小脑自动管理 |
| 复杂度控制 | 随模块增加线性增长 | 标准接口契约 + 反熵机制，O(n) 扩展 |
| 进化验证 | 无 | 双评估架构（Operational Critic + Meta Judge 物理隔离） |

### 1.8 目标用户

需要 AI 管家处理复杂多步骤任务的个人用户和团队——不满足于简单问答，期望 AI 能自主完成研究、规划、执行、审查的完整工作流，并且越用越强、越用越懂你。

### 1.9 项目分类

- **类型**：智能管家应用 + 后端 AI 组织引擎
- **领域**：AI 多 Agent 系统
- **复杂度**：高
- **架构**：前后端分离，后端核心 C++，前端待定
- **项目状态**：成熟设计（v1-v4 + A2A + 高层概念），全新实现

## 2. 成功标准

### 2.1 用户成功

| 指标 | 目标 | 验证方式 |
|------|------|----------|
| 重复任务加速 | 同类任务第二次执行耗时降低 ≥50% | 对比同类任务首次与后续执行的端到端时间 |
| 偏好学习准确率 | 用户模型推断的偏好被用户接受 ≥80% | 追踪偏好推断被采纳 vs 被覆盖的比例 |
| 主动提醒有效率 | 主动提醒中用户认为有价值的 ≥70% | 用户对提醒的反馈信号（采纳/忽略/关闭） |
| 复杂任务完成率 | 多步骤任务（研究→规划→执行→审查）端到端自主完成 ≥60% | 无需人工干预即完成的任务占比 |
| 一次性完成率 | 同类任务经经验积累后首次尝试即满足用户需求 ≥80% | 对比有/无经验模板时用户首次反馈满意的比例 |
| 用户信任度 | 用户愿意将 L2 及以上风险任务交给管家处理 | 任务风险等级分布随时间上移 |

### 2.2 系统进化成功（3 个月里程碑）

| 指标 | 目标 | 验证方式 |
|------|------|----------|
| Toolization 闭环 | 至少 1 个任务完成从涌现→模式识别→编译→替换的完整循环 | Compiler 日志 + 前台调用记录 |
| 编译产物可用 | 编译生成的插件通过 Meta Judge 盲测且投入使用 | A/B 测试结果 + 稳定区注册记录 |
| 进化可审计 | 每次进化的完整决策链可从 Event Log 复现 | 审计回放测试 |
| 系统稳定性 | 连续运行 30 天无需人工干预修复 | 运维日志 |
| 反熵触发 | 至少 1 次熵指标超阈值并成功触发自动压缩 | Entropy Monitor 日志 |

### 2.3 涌现智能验证（12 个月里程碑）

| 指标 | 目标 | 验证方式 |
|------|------|----------|
| 能力自主增长 | 系统自动编译 ≥10 个可用插件，无需开发者手动创建 | 插件注册表 + 来源追溯 |
| 组织复杂度可控 | 插件数量增长但熵指标保持在警戒线以下 ≥75% 时间 | Entropy Monitor 长期趋势 |
| 跨域能力迁移 | 在 A 领域学到的协作模式被成功应用到 B 领域 ≥1 次 | Compiler 编译记录 + 跨域调用日志 |
| 探索-稳定平衡 | 探索区任务占比稳定在 5%-15% | 双区任务分布统计 |
| 用户感知 | 用户主观感受"管家越来越聪明"，而非"功能越来越多" | 定期用户反馈 |

### 2.4 技术成功

| 指标 | 目标 | 验证方式 |
|------|------|----------|
| 通信效率 | 混合协议相比全量 context 传递压缩比 ≥10x | 消息大小对比 |
| 模块接入成本 | 新 Agent 满足标准接口契约后即可接入，无需修改 Router | 新模块接入流程记录 |

| 记忆效率 | 小脑管理的记忆摘要占 LLM context ≤15% | Context 使用统计 |

## 3. 用户旅程

### 旅程 1：主用户 — 复杂任务首次完成（成功路径）

**人物：** bender，一个技术背景的开发者，日常需要处理大量跨领域的研究、规划和执行任务。

**用户画像（系统观察）：** EvoClaw 通过持续观察已提炼出结构化画像：bender 通常在周一上午做技术调研，偏好表格对比而非长文叙述，对性能指标比价格更敏感，调研后通常会紧接着写实现方案。

**开场：** bender 需要完成一个复杂任务——调研某个技术方向，对比方案，输出一份结构化的技术选型报告。以前他要自己在多个工具间切换，手动整理信息，耗费大半天。

**上升：** 他向 EvoClaw 发出指令："帮我调研 X 技术方向，对比主流方案，输出选型报告。" Fast Router 判断这是未知的复杂任务，将其抛给后台。Dynamic Holons 自动组队——规划者拆解任务步骤，分析师并行调研各方案，审查员交叉验证结论。整个过程中，小脑通过混合协议协调通信，bender 只看到进度摘要。管家不需要追问输出格式——用户模型已经知道他要表格对比。

**高潮：** 20 分钟后，管家返回一份结构化的选型报告。报告格式恰好是他喜欢的风格——因为用户模型已经从历史交互中学到了他的偏好。更进一步，管家主动附上一句："要不要我接着起草实现方案的框架？"——因为情景记忆显示他上次调研完就做了这件事。管家比他自己更了解他的工作节奏。

**结局：** bender 给出正面反馈。这次任务的完整执行轨迹被记录到 Event Log，成为未来进化的数据源。

**揭示的需求：** Facade 统一交互、Holons 动态组队、混合协议通信协调、用户模型偏好学习、情景记忆行为预判、进度反馈机制

### 旅程 2：主用户 — 重复任务加速（进化体现）

**开场：** 一周后的周一早上，bender 还没开口，管家就主动提醒："上周你提到想调研 Y 方向，要现在开始吗？"——这来自工作记忆中上周对话的压缩提炼，加上用户模型中"周一上午做调研"的行为模式。

**上升：** bender 确认后，Fast Router 识别到这类任务已经被 Compiler 编译为标准插件——上次 Holons 的协作模式已经被降维固化。

**高潮：** 这次不再需要 Holons 临时组队。编译后的插件直接执行，5 分钟内返回报告。bender 明显感受到"管家变快了"。

**结局：** 这就是 Toolization Loop 的用户感知——同类任务从昂贵的 System 2 深度推理，进化为 System 1 秒回。原本占用的集群算力被释放，去面对下一个未知难题。

**揭示的需求：** Compiler 模式识别与编译、Fast Router 插件匹配、编译产物质量验证、用户可感知的加速体验、工作记忆驱动的主动提醒


### 旅程 3：主用户 — 边界情况（任务失败与恢复）

**开场：** bender 要求管家执行一个涉及外部 API 调用的任务，但目标 API 近期不稳定。

**上升：** 实际上在任务执行前，管家就已经发出预警："这个 API 最近 3 次调用中有 2 次返回异常数据（来自情景记忆），建议同时准备备选数据源。" bender 同意后，系统并行调用主备两个数据源。主数据源果然返回异常，Operational Critic 标记其输出质量不达标。

**高潮：** 由于预判准备了备选方案，系统自动切换到备用数据源完成任务。管家告知 bender："主数据源异常，已自动切换备选完成任务。" 这次失败被 Challenge Curator 收录到挑战集，下次遇到类似场景系统会直接采用并行策略。

**结局：** 记忆驱动的风险预判将"事后补救"变为"事前预防"。失败经验成为进化的养料。

**揭示的需求：** Operational Critic 质量把关、情景记忆风险预判、失败检测与自动恢复、Challenge Set 失败收录、并行执行策略

### 旅程 4：系统管理者 — 监控与治理

**开场：** bender 想了解系统的进化状态——哪些能力被编译了，熵指标如何，有没有异常。

**上升：** 他打开管理面板（或通过管家查询），看到系统仪表盘。管家知道 bender 查看系统状态时最关心熵指标和编译成功率（从历史查询模式中学到），所以仪表盘默认把这两个指标置顶。

**高潮：** 他发现某个插件的使用率持续下降，熵指标接近警戒线。系统已经自动标记了这个插件为"待退役"候选，并建议与另一个功能重叠的插件合并。

**结局：** bender 确认合并建议，系统执行反熵操作。组织变得更精简高效。

**揭示的需求：** 系统仪表盘、熵指标可视化、进化时间线、插件生命周期管理、反熵操作确认流程、管理界面个性化布局

### 旅程 5：开发者 — 模块贡献（开源后）

**开场：** 一个外部开发者想为 EvoClaw 贡献一个新的能力模块——比如一个专业的财务分析 Agent。

**上升：** 他按照标准接口契约规范实现模块接口（capability_profile、airbag_spec、health_probe 等），提交到探索区。

**高潮：** 模块在探索区经过 20 次任务验证，通过 Contract Validator 校验和 Meta Judge 盲测，自动晋升到稳定区。Router 无需任何修改就能路由到新模块。

**结局：** 新能力零摩擦接入，组织能力扩展。这就是标准化接口带来的 O(n) 扩展。

**揭示的需求：** 标准接口契约规范与文档、Contract Validator、探索区→稳定区晋升流程、开发者文档与 SDK

### 旅程 6：主用户 — 真实世界任务执行（系统控制）

**开场：** bender 需要在本地环境中完成一系列运维操作——部署一个新服务、配置定时备份、监控进程状态。以前他要在终端和多个工具间反复切换。

**上升：** 他向管家发出指令："帮我把 X 服务部署到本地，配置每天凌晨 3 点自动备份数据库，然后监控它的运行状态。" Fast Router 将任务拆解后分发：文件操作 Agent 准备配置文件和目录结构，Shell 执行 Agent 运行部署脚本，定时任务 Agent 创建 cron 规则。每一步高风险操作（如写入系统配置、创建 cron）都弹出确认请求，bender 逐一审批。

**高潮：** 部署完成后，管家自动启动进程监控，并告知："服务已启动，PID 12345，CPU 使用 3%，内存 128MB。定时备份已配置，下次执行时间凌晨 3:00。" 所有操作步骤和系统命令被完整记录到 Event Log，支持审计回溯。

**结局：** bender 不再需要记住复杂的命令序列。管家将运维操作从"人工逐步执行"变为"审批式自动化"。下次类似部署任务，Compiler 可能将这套流程编译为标准插件。

**揭示的需求：** 文件系统操作、Shell 命令执行、进程管理、定时任务管理、网络操作、代码执行、分级权限确认、操作审计记录

## 4. 领域需求

### 4.1 进化安全性

- **自指悖论防护**：Maintainer（Observer + Evolver）不可自动替换，只能由用户显式操作触发替换
- **评估独立性**：Meta Judge 与 Operational Critic 物理隔离，不共享上下文，A/B 裁决采用盲测
- **进化漂移控制**：宪法定期审查，Observer 健康报告包含意图对齐度评分
- **进化预算制度**：每个周期的插件新增额度、原语新增额度、探索资源比例均有上限

### 4.2 LLM 依赖管理

- **模型无关性**：系统设计不绑定特定 LLM 提供商，通过标准化接口适配不同模型
- **Token 成本控制**：per-agent token 追踪 + 全局预算上限，进化过程本身受预算约束
- **幻觉风险缓解**：Operational Critic 把关输出质量，关键约束由形式化验证引擎验证，不依赖 LLM 判断

### 4.3 数据与隐私

- **用户数据最小化**：用户模型只存储结构化偏好，永不存储原始对话
- **不可变审计**：事件日志为 append-only，所有行为、决策、状态变更均记录为不可变事件
- **记忆生命周期**：工作记忆会话结束后压缩提炼并销毁原始记录，记忆压缩过程可控

### 4.4 系统可靠性

- **故障隔离**：忒休斯之船架构，任何组件故障不导致整体崩溃，支持热替换
- **探索区隔离**：探索区模块不可直接面向用户，必须经晋升流程进入稳定区
- **通信总线高可用**：组织通信总线作为通信中枢需要高可用设计，避免单点故障

### 4.5 进化可审计性

- **完整决策链**：每次进化的提案、测试、裁决、发布全链路可从 Event Log 复现
- **零黑盒**：不允许任何不可追溯的进化步骤
- **宪法变更门控**：宪法修改需用户显式确认，设置冷静期

## 5. 创新与新颖模式

### 5.1 检测到的创新领域

**1. Toolization Loop — 组织级自动编译**
- 现有多 Agent 系统都是静态编排（LangGraph、CrewAI、AutoGen），能力边界由开发者预设
- EvoClaw 的 Compiler 能自动从多 Agent 协作轨迹中识别模式并编译为轻量插件——这是"组织自动生成工具"，目前没有同类实现
- 类比：从手工作坊到自动化工厂的跃迁

**2. 混合通信协议 — 渐进式形式化**
- 现有 Agent 通信要么纯自然语言（信息衰减），要么纯结构化（灵活性差）
- 混合态设计（形式约束由验证引擎验证 + 语义约束由 LLM 评估 + 渐进式演化）是一个新的协议范式
- 灵感来源：Logitext 论文（arXiv:2602.18095）将 LLM 推理作为 SMT 理论

**3. 仿脑分层记忆 + 小脑自动管理**
- 现有 Agent 记忆方案要么是简单的 RAG，要么是全量 context 塞入
- 六层记忆（Event Log / 工作记忆 / 情景记忆 / 语义记忆 / 用户模型 / 挑战集）+ 小脑自动管理 + 预判能力，是对人脑记忆结构的系统性借鉴
- 关键创新：记忆不只是存储，而是驱动预判和主动服务

**4. 反熵自律 — 组织主动瘦身**
- 所有现有系统只会膨胀，不会收缩
- 熵指标监控 + 自动触发压缩/合并/退役，是"组织自我代谢"的机制
- 解决长期运行系统的复杂度失控问题

### 5.2 竞争格局

| 框架 | 多 Agent 协作 | 自进化 | 工具化编译 | 记忆体系 | 反熵 |
|------|:---:|:---:|:---:|:---:|:---:|
| LangGraph | ✓ 静态图 | ✗ | ✗ | 基础 | ✗ |
| CrewAI | ✓ 角色分工 | ✗ | ✗ | 基础 | ✗ |
| AutoGen | ✓ 对话式 | ✗ | ✗ | 基础 | ✗ |
| OpenClaw | ✓ 3 Agent | ✗ | ✗ | 简单持久化 | ✗ |
| **EvoClaw** | **✓ 动态 Holons** | **✓ 制度化** | **✓ Compiler** | **✓ 仿脑六层** | **✓ 熵监控** |

### 5.3 验证方法

| 创新点 | 验证方式 | 回退方案 |
|--------|----------|----------|
| Toolization Loop | 3 个月内验证 1 次完整闭环（涌现→编译→替换） | 退化为手动插件开发，Compiler 作为辅助建议工具 |
| 混合通信协议 | 对比纯 NL 通信的压缩比和任务成功率 | 退化为结构化 JSON 协议，放弃渐进式形式化 |
| 仿脑记忆 | 对比有/无预判能力的用户满意度和任务效率 | 退化为标准 RAG + 简单用户偏好存储 |
| 反熵机制 | 12 个月内观察熵指标趋势，验证至少 1 次自动压缩 | 退化为手动定期清理 |

### 5.4 风险与缓解

| 风险 | 缓解策略 |
|------|----------|
| Compiler 编译质量不足 | Meta Judge 盲测 + 沙盒验证 + 灰度发布，不合格不进入稳定区 |
| NLTC 形式化开销大于收益 | 渐进式设计——先纯语义，逐步升级形式化等级，随时可停在任意等级 |
| 记忆预判产生错误推断 | 预判以建议形式呈现，不自动执行；用户反馈修正用户模型 |
| 反熵过度压缩导致能力丢失 | 退役前使用率验证 + 冷静期，退役模块归档可恢复 |

## 6. 项目类型需求

### 6.1 对外接口

**用户交互渠道（MVP）：**
- **Discord Bot**：主要对话入口，自然语言交互 + 结构化指令
- **Web 管理面板**：配置、监控、系统状态可视化

**结构化指令体系：**
- 系统状态查询（如 `/status`、`/health`）
- 模式配置（如 `/config`）
- 进化状态检查（如 `/evolution`、`/entropy`）
- 插件管理（如 `/plugins`）

**Facade 统一人格：** 无论 Discord 还是 Web，用户感知到的是同一个管家人格

### 6.2 内部架构

**进程模型：**
- 四大组件（Fast Router / 消息总线 / Holons / Compiler）按部署单元划分
- Agent 间通过消息传递通信，故障隔离通过进程级或沙盒级机制保证

**消息中间件（MVP: ZeroMQ，Phase 2 评估 DDS）：**
- 组件间通过消息中间件通信，支持发布/订阅和服务调用模式
- 与忒休斯之船架构高度契合——节点可独立启停、热替换
- MVP 采用 ZeroMQ（Pub-Sub + DEALER-ROUTER），Phase 2 评估 DDS 升级路径

**故障隔离：** 进程级隔离 + 沙盒级隔离，组件故障不影响系统整体

### 6.3 LLM 接入

- 通过 HTTP API 调用 LLM（OpenAI 兼容接口）
- 模型无关性：标准化 LLM 客户端接口，可适配不同提供商
- per-agent token 追踪 + 全局预算控制

### 6.4 数据持久化

- **Event Log**：MVP 采用 JSON Lines 文件（per-process append-only），Phase 2 评估 ElasticSearch
- **记忆层**：MVP 采用文件系统存储（per-type 目录），Phase 2 评估向量数据库
- 框架层面预留存储抽象接口，不绑定具体实现，后续可替换

### 6.5 多用户

- MVP 单用户
- 框架层面预留多用户隔离点：用户模型、记忆层、Event Log 按用户 namespace 隔离

## 7. 范围定义与分阶段开发

### 7.1 MVP 策略

**策略：最小闭环验证（Toolization Loop 跑通）**

MVP 的核心目标不是做一个功能完善的管家，而是验证"组织能自我进化"这个核心假设。最小可验证路径：

> 用户发出复杂任务 → Holons 协作完成 → Compiler 识别模式 → 编译为插件 → 下次同类任务由 Router 秒回

### 7.2 Phase 1: MVP（3 个月）— 闭环验证

**Must-Have：**

| 组件 | MVP 范围 | 简化策略 |
|------|----------|----------|
| Fast Router | 工具库匹配 + 未知任务转发 | 规则匹配，不需要 ML |
| AI-IPC Bus | 消息中间件基础通信（发布/订阅 + 服务调用） | MVP 采用 ZeroMQ，单机部署 |
| Dynamic Holons | 3 个基础 Agent（规划者、执行者、审查员）临时组队 | 固定角色，不需要动态发现 |
| Compiler | 模式识别 + 基础编译（协作轨迹→脚本插件） | 人工辅助确认编译结果 |
| NLTC 协议 | 基础版（形式约束 + 语义约束，FL-0 到 FL-2） | 不需要完整渐进式演化 |
| 记忆层 | Event Log（ES）+ 工作记忆 + 用户模型（向量库） | 情景/语义记忆简化为基础检索 |
| Facade | Discord Bot + 统一人格 | 单渠道 |
| Web 面板 | 基础监控（系统状态、进化日志） | 只读面板 |
| 结构化指令 | `/status`、`/health`、`/plugins` | 最小指令集 |

**Nice-to-Have（Phase 1 可手动替代）：**
- 主动提醒和预判
- 反熵机制
- Meta Judge 盲测（人工审查编译结果即可）

### 7.3 Phase 2: 成长（6 个月）— 智能增强

- 完整分层记忆体系（情景记忆 + 语义记忆 + 挑战集）
- 记忆驱动的预判和主动服务
- Operational Critic + Meta Judge 双评估架构
- Compiler 全自动化
- NLTC 渐进式形式化（FL-0 到 FL-4 完整生命周期）
- 反熵机制原型（熵指标监控 + 手动触发压缩）
- Web 面板增强（配置管理、进化时间线可视化）

### 7.4 Phase 3: 扩展（12 个月）— 涌现智能

- 完整反熵自律（自动触发压缩/合并/退役）
- 进化预算制度 + 宪法审查
- 探索区/稳定区双区完整实现
- 跨域能力迁移验证
- 多渠道接入（Telegram/飞书/微信）
- UMI 契约规范 + 开发者 SDK
- 开源准备

### 7.5 关键风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| DDS 在 AI 场景下性能不足 | 组件通信瓶颈 | MVP 采用 ZeroMQ 验证，预留通信抽象层可替换为 DDS |
| Compiler 编译质量不可控 | 核心假设无法验证 | Phase 1 人工辅助确认，降低全自动要求 |
| C++ 开发周期长 | MVP 交付延迟 | 聚焦最小闭环，非核心功能推迟 |
| 单人开发资源有限 | 范围蔓延 | 严格按 Phase 划分，Phase 1 只做 Must-Have |

### 7.6 MVP 验收标准

1. ✅ 用户通过 Discord 发出复杂任务，Holons 协作完成
2. ✅ Compiler 识别至少 1 个重复模式并编译为插件
3. ✅ 编译后的插件被 Router 成功调用，同类任务明显加速
4. ✅ Event Log 完整记录全链路，可审计回放
5. ✅ 系统连续运行 7 天无需人工干预修复
